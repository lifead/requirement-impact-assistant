# AGENTS.md - оперативные правила проекта

## Назначение проекта

**requirement-impact-assistant** - MVP-прототип программы анализа влияния проектных запросов, связанных с изменениями требований, функциональности, API, интеграций и проектных решений.

Программа предназначена для диссертационного исследования по специальности 2.3.4 "Управление в организационных системах" и потенциальной регистрации как программа для ЭВМ.

Ключевая позиция проекта: LLM-ассистент не является субъектом управления и не принимает управленческие решения. Он формирует предварительный аналитический материал, а экспертное заключение и управленческое рассмотрение остаются за человеком.

Проект ведется как staged workflow: от сырой идеи и требований до технического проекта, implementation planning, реализации, тестов, ревью и документации.

## Workflow feature

Крупная feature проходит этапы последовательно:

1. Сырая постановка задачи.
2. Первичный предпроектный анализ.
3. Фиксация уточнений и решений.
4. Черновик требований.
5. Ревью требований.
6. Технический проект.
7. Ревью технического проекта.
8. План реализации.
9. Ревью плана реализации.
10. Реализация отдельными task.
11. Проверка и ревью изменений.

## Phase gates

- Нельзя переходить к требованиям без завершенного первичного анализа и зафиксированных уточнений.
- Нельзя переходить к техническому проектированию без утвержденных требований.
- Нельзя переходить к плану реализации без завершенного и проверенного технического проекта.
- Нельзя переходить к реализации без завершенного плана реализации и явного решения о начале конкретной task.
- На ранних этапах не фиксируются преждевременные технические решения: БД, структура таблиц, endpoint-ы, архитектурный стиль, фреймворки.
- Реализация выполняется маленькими task: одна task - один проверяемый результат - один review - один commit.
- До явной команды пользователя на реализацию конкретной task код, project files и тесты не создаются.

## Правила работы

- **Код не создается до отдельного явного решения.** Решение о начале реализации принимается после завершения требований, технического проекта, плана реализации и review.
- **Все изменения выполняются маленькими шагами.** Один шаг - одна законченная мысль или операция.
- **Commit выполняется только по явной команде пользователя.** Без команды "commit" изменения не фиксируются.
- **Перед каждым commit необходимо показать `git diff --stat`**, чтобы пользователь видел сводку изменений.
- **Интеллектуальный анализ выполняется только через application-level boundary.** UI, controllers, pages и бизнес-логика не должны вызывать LLM напрямую; все обращения идут через `IAiAnalysisEngine` или согласованный аналог.
- **LLM-интеграция проектируется через provider boundary.** DeepSeek рассматривается как первый планируемый real provider, но секреты не хранятся в репозитории; demo/mock provider используется для локальной демонстрации и воспроизводимости.
- **RAG и внешние AI/RAG-движки добавляются только по отдельному решению.** Не добавлять RAG, embeddings, rerank, Dify-интеграцию, agentic workflow или аналоги без явной команды пользователя.
- **MVP не должен разрастаться в Jira/Confluence/ALM/RAG/workflow-платформу.** Ограничения MVP не должны закрывать будущие точки расширения.
- **Экспертное решение отделяется от результата LLM/AI.** LLM/AI/RAG формирует предварительный аналитический материал; экспертное решение или заключение фиксирует человек.
