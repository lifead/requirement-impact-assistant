# MVP1-STAB-001. Автоматическое восстановление SQLite-схемы при старте

Статус: реализовано в текущем рабочем diff.

Severity / area: Medium / SQLite persistence, startup.

## Проблема

При запуске приложения с отсутствующей SQLite-базой файл базы мог быть создан, но прикладная схема хранения не гарантированно восстанавливалась до первого обращения к данным.

Риск: локальная демонстрация и smoke-проверка становились зависимыми от ручной подготовки базы или предыдущего состояния рабочей директории.

## Причина

Инициализация persistence была подключена как регистрация сервисов, но на старте приложения не выполнялся явный application-level шаг применения схемы SQLite.

## Scope decision

В рамках stabilization фиксируется только восстановление локальной SQLite-схемы при старте приложения.

Не входит:

- изменение доменной модели;
- изменение production workflow анализа;
- добавление RAG, внешних интеграций или seed-данных;
- автоматическое создание пользовательских аналитических записей.

## Внесенное решение

В текущем diff приложение вызывает `ApplyApplicationPersistenceMigrationsAsync` при старте.

Регистрация persistence дополнена extension-точкой для применения миграций через `ServiceCollectionExtensions`.

Проверка зафиксирована тестом: при отсутствующей SQLite-базе стартовая конфигурация создает схему, после чего коллекция `Analyses` доступна и остается пустой.

## Seed/reference-data вывод

Для этого item seed/reference data не требуется.

Восстановление схемы не должно создавать демонстрационные анализы, экспертные заключения или иные пользовательские записи. Пустая таблица `Analyses` после создания схемы является ожидаемым результатом.

## Проверки

- Missing SQLite database -> schema created -> `Analyses` empty.
- Application startup вызывает application-level migration boundary, а не прямую инициализацию из UI/pages.
- Решение не меняет LLM/provider boundary и не добавляет внешние AI/RAG-зависимости.
