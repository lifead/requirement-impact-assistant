# MVP-4 UI Information Architecture Analysis

## Статус

Этот документ является analysis artifact для MVP-4.

Документ не является требованиями, техническим проектом, implementation plan или решением о начале реализации.

Документ не фиксирует обязательную UI-модель, layout, PageModel, routes, components, storage/API/DB/migrations/state-machine, RAG pipeline или изменения кода.

## Назначение

Цель анализа - рассмотреть текущие экраны и смысловую структуру MVP-4 UI до перехода к требованиям.

MVP-4 направлен на упрощение пользовательского маршрута и визуальной иерархии вокруг одного сохраненного `analysis` без изменения предметной логики программы.

Результат этого документа - наблюдения, риски и открытые вопросы для следующего SDD-шагa: `mvp4-ui-clarifications.md`.

## Источники Анализа

Использованы следующие фактические источники:

- UI explorer report по текущим Razor Pages:
  - `Pages/Index.cshtml`;
  - `Pages/Analyses/Index.cshtml`;
  - `Pages/Analyses/Details.cshtml`;
  - `Pages/Analyses/Review.cshtml`;
  - `Pages/Analyses/ExpertEvaluation.cshtml`.
- Docs explorer report по MVP-4 concept, MVP-3 demo scenario и project SDD rules.
- Проектные правила:
  - LLM/AI/RAG формирует предварительный аналитический материал;
  - экспертное заключение и управленческое рассмотрение остаются за человеком;
  - phase gates сохраняются: `analysis -> clarifications -> requirements -> requirements review -> technical design -> design review -> implementation plan -> plan review -> small implementation tasks`.

## Текущие Экраны И Смысловые Блоки

### `Pages/Index.cshtml`

Стартовая страница приложения.

На уровне данного анализа экран рассматривается как входная точка в приложение, а не как основной рабочий экран одного сохраненного `analysis`.

### `Pages/Analyses/Index.cshtml`

Экран списка сохраненных анализов.

Смысловые блоки:

- список сохраненных `analysis`;
- краткая информация по каждому анализу;
- действия `Review` и `Details`.

Наблюдение: на этом экране уже появляется развилка пользовательского маршрута. Требуется уточнить, как пользователь должен понимать различие между переходом к `Review` и переходом к `Details`.

### `Pages/Analyses/Review.cshtml`

Экран проверки входных данных и запуска preliminary analysis.

Смысловые блоки:

- минимальная готовность входных данных;
- выбор режима `DirectLlm` / `ExternalRag`;
- запуск анализа;
- исходные поля анализа;
- `manual context fragments`.

Наблюдение: экран выглядит более сфокусированным, чем `Details`, но действие запуска анализа расположено рядом с большим объемом исходных данных и контекста. Для clarifications важно определить, какой главный пользовательский вопрос решает этот экран.

### `Pages/Analyses/Details.cshtml`

Основной перегруженный экран текущего UI.

Смысловые блоки:

- сценарий анализа;
- входные данные;
- `expert conclusion`;
- `preliminary AI result`;
- metadata;
- grounds/limitations;
- impact map;
- diagnostics;
- raw response;
- input snapshot;
- retrieved context;
- expert evaluation status;
- export;
- manual context.

Наблюдение: `Details` содержит почти весь жизненный цикл анализа на одном длинном экране. Это затрудняет различение того, что является исходными данными, что является предварительным AI-материалом, что является диагностикой, а что относится к экспертному действию человека.

### `Pages/Analyses/ExpertEvaluation.cshtml`

Экран человеческой оценки результата AI.

Смысловые блоки:

- context summary;
- input data;
- grounds/limitations;
- preliminary impact map;
- retrieved context;
- warnings/limitations;
- оценки элементов карты;
- missed items;
- corrections;
- общие оценки.

Наблюдение: экран содержит много таблиц и полей. `AI-result` и `expert-input` находятся рядом в одной рабочей зоне. Это допустимо как предмет анализа, но требует проверки, достаточно ли явно разделены предварительный материал AI и экспертная оценка человека.

## Наблюдения По Перегрузке

Основная перегрузка находится в `Details`.

Причина перегрузки не только в количестве блоков, но и в том, что на одном экране собраны разные смысловые роли:

- исходные данные;
- контекст, добавленный человеком;
- retrieved context;
- preliminary AI material;
- grounds/limitations;
- diagnostics;
- expert evaluation status;
- expert conclusion;
- export.

`ExpertEvaluation` также выглядит плотным экраном, потому что одновременно показывает материал для оценки и поля экспертной оценки.

`Review` выглядит более понятным, но требует уточнения роли: это экран проверки входа, экран запуска анализа или комбинированный шаг.

На текущем этапе нельзя делать вывод о конкретном UI-решении. Можно зафиксировать только проблему анализа: текущая структура требует явного разделения смысловых ролей экранов и блоков.

## Разделение Слоев

Для MVP-4 важно сохранить различимость следующих слоев:

- `input`;
- `manual context`;
- `retrieved context`;
- `preliminary AI material`;
- `expert evaluation`;
- `expert conclusion`.

### `input`

К этому слою относятся текущее состояние, проектное изменение, причина и источник изменения.

В текущем UI входные данные присутствуют в `Review`, `Details` и `ExpertEvaluation`. Для clarifications нужно определить, где пользователь должен видеть входные данные как основной объект проверки, а где только как контекст.

### `manual context`

`Manual context` присутствует в `Review` и `Details`.

Риск: пользователь может не отличить ручной контекст, добавленный человеком, от retrieved context, найденного AI/RAG-контуром.

### `retrieved context`

`Retrieved context` присутствует в `Details` и `ExpertEvaluation`.

Риск: retrieved context может восприниматься как равнозначный экспертному основанию, если рядом нет ясного указания на его предварительный и вспомогательный характер.

### `preliminary AI material`

`Preliminary AI result` и preliminary impact map присутствуют в `Details` и `ExpertEvaluation`.

Риск: preliminary AI material может быть визуально воспринят как итоговое заключение программы, если он расположен рядом с expert conclusion без явного смыслового разделения.

### `expert evaluation`

`ExpertEvaluation` содержит оценки элементов карты, missed items, corrections и общие оценки.

Риск: экспертная оценка может смешиваться с AI-result, если экран не помогает понять, где заканчивается материал AI и где начинается действие эксперта.

### `expert conclusion`

`Expert conclusion` присутствует в `Details`.

Ключевой риск: expert conclusion нельзя смешивать с preliminary result. Для следующего шага нужно уточнить, где и как пользователь должен понимать статус экспертного заключения как человеческого действия.

## Проверка Демосценария MVP-3

Основной проверочный сценарий: изменение уведомления заявителя после смены статуса заявки в демонстрационной системе обработки заявок.

Входы сценария:

- текущее состояние;
- проектное изменение;
- причина;
- источник изменения.

Для этого сценария UI должен позволять различить:

- что сейчас существует в системе;
- что предлагается изменить;
- какой контекст добавлен человеком;
- какой контекст был найден или не найден AI/RAG-контуром;
- какой preliminary result сформирован;
- какую оценку дает эксперт;
- какое экспертное заключение или решение остается за человеком.

Текущие экраны содержат необходимые смысловые блоки, но они распределены и частично пересекаются между `Review`, `Details` и `ExpertEvaluation`.

Открытый вопрос для clarifications: какой экран является главным рабочим местом пользователя при разборе одного сохраненного `analysis` в рамках демосценария.

## Открытые Вопросы Для Clarifications

1. Какой экран является главным рабочим экраном одного сохраненного `analysis`?

2. Какой главный вопрос должен решать каждый текущий экран:
   - `Analyses/Index`;
   - `Review`;
   - `Details`;
   - `ExpertEvaluation`?

3. Как пользователь должен различать `Review` и `Details` в списке анализов?

4. Где пользователь должен проверять входные данные сценария?

5. Где пользователь должен видеть preliminary AI material как предварительный аналитический материал, а не как итоговое решение программы?

6. Где пользователь должен видеть grounds/limitations так, чтобы они не смешивались с expert conclusion?

7. Как различать `manual context` и `retrieved context` на уровне смысла?

8. Какие блоки критичны для демонстрационного сценария MVP-3, а какие являются вспомогательными или диагностическими?

9. Нужно ли сохранять текущую роль `Details` как экрана, где собран почти весь жизненный цикл анализа, или эта роль требует уточнения на уровне требований?

10. Достаточно ли явно в текущем UI отделены:
    - preliminary AI material;
    - expert evaluation;
    - expert conclusion?

11. Какие данные пользователь должен видеть сразу при открытии одного `analysis`, а какие могут быть предметом последующего просмотра?

12. Какие диагностические материалы нужны пользователю в основном сценарии, а какие нужны только для проверки корректности работы MVP?

## Red Flags Для Следующего Этапа

На следующем этапе нельзя допустить следующие ошибки:

- AI/RAG/LLM начинает выглядеть как субъект управленческого решения.
- `expert conclusion` смешивается с `preliminary AI result`.
- `manual context` и `retrieved context` не различаются.
- Grounds/limitations исчезают или воспринимаются как техническая диагностика без смысловой роли.
- Отсутствие retrieved context маскируется и не видно пользователю.
- Диагностические блоки начинают определять основной пользовательский маршрут.
- Возникают скрытые решения по UI layout, PageModel, routes, components, storage/API/DB/migrations/state-machine.
- Возникает скрытое проектирование RAG pipeline, embeddings, rerank или retrieval architecture.
- MVP-4 расширяется в task board, approval workflow, dashboard, PDF/export feature, ALM/Jira/Confluence-аналог или RAG-платформу.
- `primary / secondary / diagnostic` превращается в обязательную модель до завершения анализа и clarifications.
- Clarifications начинают формулироваться как requirements или implementation tasks.

## Вывод И Следующий SDD-Шаг

Текущий UI уже содержит основные смысловые блоки, необходимые для демонстрационного сценария MVP-3 и для регистрационно безопасного разделения AI-материала и экспертного действия.

Главная проблема текущей структуры - не отсутствие данных, а перегрузка и смешение смысловых ролей на экранах `Details` и `ExpertEvaluation`.

Для перехода к требованиям пока недостаточно зафиксировать общий принцип упрощения UI. Сначала нужно уточнить роли экранов, различимость слоев и состав смысловых блоков, критичных для демонстрационного сценария.

Следующий SDD-шаг: подготовить `mvp4-ui-clarifications.md`.

Этот следующий документ должен зафиксировать ответы на открытые вопросы данного анализа и не должен переходить к technical design, implementation plan или изменениям кода.
