# MVP-4 UI simplification concept

## Назначение документа

Документ фиксирует первичную концепцию MVP-4 для упрощения пользовательского интерфейса программы анализа влияния проектных запросов.

Это концептуальный SDD-артефакт. Он не является требованиями, техническим проектом, implementation plan, task breakdown или решением о начале реализации. Код, project files, tests, migrations, новые компоненты UI и изменения маршрутов не создаются на основании этого документа без отдельного последующего этапа и явной команды пользователя.

## Основание

MVP-4 начинается после завершения MVP-3 и merge ветки `mvp3` в `main`.

Основанием для концепции являются:

- `docs/program/mvp0/initial-concept.md`;
- `docs/program/mvp0/clarification-decisions.md`;
- `docs/program/mvp0/ui-concept.md`;
- `docs/program/mvp3/mvp3-ux-registration-strategy.md`;
- `docs/program/mvp3/mvp3-ui-flow.md`;
- `docs/program/mvp3/mvp3-requirements-draft.md`;
- `docs/program/mvp3/mvp3-demo-smoke-checklist.md`;
- текущая реализация Razor Pages UI в `src/RequirementImpactAssistant.Web/Pages/Analyses/`;
- результаты read-only review с субагентами по текущим экранам, демонстрационному сценарию и регистрационной рамке.

## Исходная позиция

Программа предназначена для поддержки человека при анализе проектного запроса или проектного изменения.

Она помогает:

1. Зафиксировать текущее состояние как точку отсчета.
2. Описать предлагаемое проектное изменение.
3. Добавить связанный ручной контекст.
4. Получить предварительную карту влияния от AI/RAG/LLM.
5. Показать основания и ограничения предварительного результата.
6. Передать материал человеку для экспертной оценки.
7. Зафиксировать экспертное заключение.
8. Получить воспроизводимый Markdown/JSON export.

Ключевая позиция сохраняется: AI/RAG/LLM не является субъектом управления и не принимает управленческих решений. Программа подготавливает аналитический материал, а экспертная оценка и экспертное заключение остаются за человеком.

## Рабочая формулировка цели MVP-4

Цель MVP-4 - упростить пользовательский маршрут и визуальную иерархию существующего приложения вокруг одного сохраненного артефакта анализа, не меняя предметную логику программы.

MVP-4 должен сделать интерфейс более пригодным для демонстрационного, исследовательского и регистрационного использования за счет того, что пользователь быстрее понимает:

- что анализируется;
- что является текущим состоянием;
- что является проектным изменением;
- какой контекст добавлен человеком;
- какой контекст найден или не найден внешним AI/RAG-контуром;
- где находится предварительный результат AI/RAG/LLM;
- где начинается человеческая экспертная оценка;
- какое экспертное заключение зафиксировано;
- какой результат можно экспортировать.

## Что является главным артефактом

Главный объект UI - один сохраненный `analysis`: анализ влияния одного проектного изменения.

Интерфейс не должен выглядеть как:

- чат с LLM;
- task board;
- backlog;
- sprint/workflow-панель;
- Jira/Confluence/ALM-аналог;
- RAG-платформа;
- средство автоматического принятия решений.

Пользовательский маршрут должен быть организован вокруг одного артефакта: от входных данных к предварительной карте влияния, затем к экспертной оценке, экспертному заключению и export.

## Принцип MVP-4

Основной принцип: на каждом экране должен быть один главный вопрос.

| Экран | Главный вопрос |
| --- | --- |
| `Index` | Какой анализ открыть или создать? |
| `Create` / `Edit` | Что анализируем? |
| `Review` | Готовы ли входные данные и в каком режиме запускать предварительный анализ? |
| `Details` | Что сейчас известно по анализу и какой следующий шаг? |
| `ExpertEvaluation` | Как эксперт оценивает элементы карты влияния? |
| `ExpertConclusion` | Какое человеческое экспертное заключение фиксируется? |
| Export actions | Как выгрузить сохраненный результат без повторного анализа? |

## Основной демонстрационный сценарий

Для MVP-4 основной проверочный сценарий остается сценарий из `docs/program/mvp3/mvp3-demo-smoke-checklist.md`: изменение уведомления заявителя после смены статуса заявки в демонстрационной системе обработки заявок.

Сценарий на уровне UI:

1. Пользователь открывает список анализов и создает новый анализ.
2. Пользователь заполняет текущее состояние.
3. Пользователь описывает проектное изменение.
4. Пользователь указывает ситуацию, источник и при необходимости добавляет ручной контекст.
5. Пользователь открывает проверку перед анализом и выбирает режим `Direct LLM` или `External AI/RAG`.
6. Пользователь запускает предварительный анализ.
7. Пользователь просматривает предварительный результат, карту влияния, основания и ограничения.
8. Эксперт выполняет экспертную оценку.
9. Эксперт фиксирует экспертное заключение.
10. Пользователь экспортирует сохраненный результат в Markdown или JSON.

На первом уровне UI сценарий должен отвечать на пять вопросов:

1. Что сейчас?
2. Что меняется?
3. Что добавил человек?
4. Что нашел или не нашел AI/RAG-контур?
5. Что решил эксперт?

## Информационная иерархия как открытый вопрос

MVP-4 должен уменьшить когнитивную нагрузку интерфейса, но на уровне концепции еще нельзя фиксировать обязательную модель экранных уровней.

Требуется отдельный промежуточный анализ, который ответит на вопросы:

- какие сведения пользователь должен видеть сразу;
- какие сведения нужны для проверки и уточнения;
- какие сведения нужны только для воспроизводимости, диагностики или review;
- как разделять текущее состояние, проектное изменение, ручной контекст, retrieved context, preliminary result, expert evaluation и expert conclusion;
- какие сведения нельзя скрывать без потери регистрационной и исследовательской ясности;
- какие сведения можно показывать менее заметно без потери смысла.

Трехуровневая модель вида `primary / secondary / diagnostic` может рассматриваться только как возможная рабочая гипотеза для последующего анализа. Она не фиксируется этим документом как обязательная структура интерфейса.

## Текущие экраны как предмет последующего анализа

Текущая реализация показывает, что пользовательская сложность распределена неравномерно.

Предварительная рабочая гипотеза:

- `Index`, `Create`, `Edit` и `ExpertConclusion` ближе к одному основному пользовательскому вопросу;
- `Review`, `Details` и `ExpertEvaluation` требуют отдельного анализа информационной структуры, потому что совмещают несколько смысловых задач;
- `Details` является ключевой точкой риска, так как может начать выглядеть как витрина всех данных сразу, а не как рабочее пространство сохраненного анализа;
- `ExpertEvaluation` является второй точкой риска, так как экспертная работа может смешаться с технической диагностикой результата;
- `Review` требует проверки как preflight-экран, чтобы не превращаться в повтор полного содержимого анализа.

Эти пункты не являются требованиями и не фиксируют будущую структуру экранов. Их назначение - определить предмет следующего SDD-документа.

## Границы MVP-4

В MVP-4 входит:

- упрощение визуальной иерархии существующих экранов;
- уточнение первичного пользовательского маршрута;
- анализ информационной иерархии без преждевременной фиксации конкретной модели уровней;
- сокращение избыточных пояснений;
- улучшение читаемости `Details`, `Review`, `ExpertEvaluation`;
- сохранение регистрационно безопасных формулировок;
- сохранение демонстрационного сценария на обезличенных данных.

В MVP-4 не входит:

- новая доменная модель;
- новая БД или migrations;
- новый workflow согласования;
- новый state machine;
- task board, backlog или sprint management;
- автоматическое принятие, отклонение или эскалация проектного изменения;
- создание задач, pull request, изменений в требованиях, документах, API или коде;
- новая RAG-платформа;
- embeddings, vector database, rerank или собственный retrieval pipeline;
- новая внешняя интеграция;
- расширение Dify adapter;
- перевод UI, PageModels, domain model или export на provider-specific DTO;
- новый export-формат;
- PDF-генерация;
- юридическая подготовка полного пакета заявки на регистрацию.

## Red flags

MVP-4 считается выходящим за границы, если в требованиях или реализации появляется хотя бы один из признаков:

- AI/RAG/LLM результат выглядит как решение программы;
- expert conclusion визуально смешивается с preliminary result;
- manual context и retrieved context перестают различаться;
- блок оснований и ограничений результата исчезает полностью;
- отсутствие retrieved context маскируется выдуманными источниками;
- provider-specific детали становятся смысловой основой UI;
- интерфейс начинает обещать полный, гарантированный или автоматический анализ;
- появляются task board, approval workflow, dashboard, PDF или ALM-интеграции;
- упрощение UI требует изменения доменной модели, хранения, AI/RAG boundary или export-формата без отдельного решения о расширении scope.

## Предварительный критерий успешности

MVP-4 можно считать успешным на концептуальном уровне, если пользователь может пройти основной демонстрационный сценарий и без пояснений разработчика понять:

- какой артефакт он создает;
- какие данные являются точкой отсчета;
- какое изменение анализируется;
- какие материалы добавлены вручную;
- какие основания вернул или не вернул внешний AI/RAG-контур;
- что является предварительной картой влияния;
- что проверяет эксперт;
- какое заключение зафиксировал человек;
- где получить воспроизводимый Markdown/JSON export.

## Следующий SDD-шаг

После review этой концепции следующий шаг - подготовить отдельный документ `MVP-4 UI information architecture analysis`.

Назначение промежуточного анализа - согласовать информационную структуру интерфейса до перехода к требованиям.

Этот analysis-документ должен определить:

- какие смысловые блоки должны быть видимы пользователю в MVP-4;
- как эти блоки соотносятся друг с другом;
- какие различия между слоями результата нельзя терять;
- какие сведения нужны в основном сценарии, а какие только при проверке или диагностике;
- какие вопросы остаются открытыми и не должны превращаться в требования преждевременно.

Этот analysis-документ не должен фиксировать:

- обязательную модель `primary / secondary / diagnostic` или любую другую заранее выбранную схему уровней;
- конкретные UI-компоненты или layout-механику;
- структуру представлений, PageModel-ов, CSS или тестов;
- изменения доменной модели, БД, migrations, AI/RAG boundary или export-формата.

Только после review `MVP-4 UI information architecture analysis` можно переходить к MVP-4 requirements draft.
