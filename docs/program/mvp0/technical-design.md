# Технический проект MVP программы анализа влияния проектных запросов

## Назначение документа

Документ фиксирует технический проект MVP программы анализа влияния проектных запросов.

Документ подготовлен после review требований и опирается на:

- `initial-concept.md`;
- `clarification-decisions.md`;
- `requirements-draft.md`;
- `ui-concept.md`.

Документ не является реализацией, implementation-plan, перечнем task, пользовательскими историями или тестовым планом. Его задача - выбрать техническую рамку MVP, основные модули, модель данных, способ интеллектуального анализа, сохранение результата и экспорт так, чтобы программа оставалась небольшой, демонстрируемой, пригодной для главы 4 диссертации и потенциальной регистрации как программа для ЭВМ.

## Цели технического проекта

Технический проект должен обеспечить:

- реализацию одного законченного контура анализа проектного запроса;
- сохранение исходных данных, связанного контекста, результата интеллектуального анализа, экспертной оценки и экспертного заключения;
- воспроизводимый просмотр сохраненного анализа;
- экспорт результата в Markdown и JSON;
- возможность использовать сохраненные и экспортированные данные в экспериментальной проверке главы 4;
- четкое разделение аналитической роли AI/RAG/LLM и экспертной роли человека.

Технический проект не должен расширять MVP до системы управления задачами, корпоративной wiki, собственной RAG-платформы или автоматического агента управления изменениями.

## Архитектурное решение

Для MVP выбирается локальное web-приложение в стиле модульного монолита.

Причины выбора:

- программа должна быть небольшой и законченной;
- все ключевые сценарии связаны с одним артефактом анализа;
- MVP не требует распределенной архитектуры, микросервисов или очередей;
- один deployable artifact проще демонстрировать, описывать в главе 4 и готовить к регистрации программы для ЭВМ;
- внутреннее модульное разделение достаточно для последующего развития без усложнения первого прототипа.

Архитектура приложения:

```text
---------------------------------------------------------------+
| Web UI                                                       |
| - анализы, контекст, интеллектуальный анализ, карта, эксперт |
+---------------------------------------------------------------+
| Application Services                                         |
| - сценарии создания анализа, analysis engine, оценка, экспорт|
+---------------------------------------------------------------+
| Domain Model                                                 |
| - Analysis, ContextFragment, ImpactMap, ExpertEvaluation     |
+---------------------------------------------------------------+
| Infrastructure                                               |
| - SQLite, file storage, AI/LLM adapters, exporters           |
+---------------------------------------------------------------+
```

Внешние корпоративные системы в MVP не подключаются. Данные из Jira, GitLab, Confluence, писем, чатов или других источников используются только как вручную введенный или предварительно загруженный текст.

## Выбранный стек

Для MVP выбирается основной стек:

- ASP.NET Core 8 Razor Pages;
- C#;
- SQLite;
- Entity Framework Core;
- локальное хранение загруженных файлов;
- серверная генерация Markdown и JSON;
- `IAiAnalysisEngine` с первой реализацией прямого LLM-анализа;
- LLM provider adapter с одной начальной реализацией.

Обоснование:

- Razor Pages достаточно для небольшого web-приложения с формами и последовательным пользовательским сценарием;
- отдельный frontend-SPA не нужен для MVP и усложнит прототип;
- SQLite достаточен для локальной демонстрации, регистрации программы и экспериментальных кейсов;
- EF Core позволяет явно описать доменную модель без ручной работы с SQL на первом этапе;
- `IAiAnalysisEngine` не позволяет UI и бизнес-логике вызывать LLM напрямую;
- provider adapter позволяет заменить LLM-провайдера без изменения analysis engine и доменной логики;
- Markdown и JSON можно генерировать на сервере из сохраненного артефакта анализа.

Не выбираются для MVP:

- React/Vue/Angular SPA;
- микросервисы;
- message broker;
- vector database;
- собственный полноценный RAG;
- внешние интеграции с Jira, GitLab, Confluence, почтой или чатами;
- сложная авторизация и ролевая модель;
- PDF-экспорт как обязательная функция.

## Основные модули

### Analysis module

Отвечает за создание и хранение анализа проектного запроса.

Функции:

- создать анализ;
- редактировать название, исходное требование, предлагаемое изменение, описание ситуации и источник изменения;
- открыть сохраненный анализ;
- показать состояние артефакта анализа.

### Context module

Отвечает за связанный проектный контекст.

Функции:

- добавить фрагмент контекста вручную;
- загрузить подготовленный Markdown, TXT или JSON-файл;
- сохранить тип фрагмента и источник;
- показать состав контекста перед запуском интеллектуального анализа;
- позволить запуск анализа без дополнительных фрагментов, если заполнены минимальные поля анализа.

### AI analysis engine module

Отвечает за запуск интеллектуального анализа через application-level интерфейс `IAiAnalysisEngine` и сохранение результата.

Функции:

- собрать входные данные анализа;
- добавить связанный контекст, если он есть;
- вызвать `IAiAnalysisEngine`;
- сохранить raw response;
- сохранить структурированный результат или состояние ошибки;
- явно маркировать результат как предварительный аналитический материал.

UI, controllers, pages и бизнес-логика не вызывают LLM напрямую. Они передают проектный запрос и подготовленные материалы в application service, который обращается к `IAiAnalysisEngine`.

Первая реализация интерфейса для MVP-0 - `DirectLlmAnalysisEngine`. Она принимает проектный запрос и добавленные материалы, формирует prompt, вызывает LLM через provider adapter и возвращает карту влияния, пригодную для экспертной оценки и сохранения.

Будущие реализации могут быть добавлены как адаптеры того же интерфейса, например `DifyAnalysisEngine`, `ExternalRagAnalysisEngine` или `CustomRagAnalysisEngine`. Они могут использовать внешний AI/RAG-движок, retrieval, rerank, embeddings, vector store или многошаговый поиск контекста, но такие возможности не входят в MVP-0 и требуют отдельного решения.

### Impact map module

Отвечает за представление структурированной карты влияния.

Функции:

- показать потенциально затронутые элементы;
- показать риски, противоречия, недостающую информацию и вопросы;
- показать варианты для экспертного рассмотрения;
- сохранить предварительную аналитическую оценку;
- при наличии сохранить ссылки результата на фрагменты контекста.

### Expert review module

Отвечает за экспертную оценку человеком.

Функции:

- отметить элементы или группы элементов как подтвержденные, скорректированные, отклоненные или требующие уточнения;
- зафиксировать существенные пропущенные элементы;
- сохранить оценку достаточности контекста;
- сохранить оценку полезности результата;
- сохранить экспертное заключение, комментарий, основание и дату фиксации.

### Export module

Отвечает за экспорт результата.

Функции:

- сформировать Markdown-отчет;
- сформировать JSON-представление;
- включить исходные данные, контекст, карту влияния, экспертную оценку и экспертное заключение;
- явно указать, что LLM не принимает управленческое решение.

### Persistence module

Отвечает за хранение данных.

Функции:

- хранить основные сущности в SQLite;
- хранить загруженные файлы в локальном каталоге приложения;
- сохранять метаданные файлов и извлеченный текст;
- обеспечивать воспроизводимый просмотр сохраненного анализа.

## Доменная модель

### Analysis

Основной артефакт программы.

Поля:

- id;
- title;
- status;
- originalRequirement;
- proposedChange;
- situationDescription;
- changeSource;
- createdAt;
- updatedAt;
- fixedAt;
- aiAnalysisResult;
- expertEvaluation;
- expertConclusion.

Имена полей доменной модели могут быть уточнены при реализации, чтобы не ограничивать анализ только изменениями требований. На уровне пользовательского смысла анализ должен поддерживать проектный запрос, где изменение требования является основным частным случаем.

### ContextFragment

Фрагмент связанного проектного контекста.

Поля:

- id;
- analysisId;
- type;
- source;
- text;
- fileName;
- filePath;
- createdAt.

Типы:

- task;
- documentFragment;
- comment;
- apiDescription;
- architecturalConstraint;
- testCase;
- previousDecision;
- other.

### AiAnalysisResult

Результат одного запуска интеллектуального анализа.

Поля:

- id;
- analysisId;
- status;
- generatedAt;
- engineName;
- providerName;
- modelName;
- promptVersion;
- inputSnapshot;
- rawResponse;
- impactMap;
- errorMessage.

`engineName`, `providerName`, `modelName` и `promptVersion` сохраняются как метаданные воспроизводимости, но не задают жесткую зависимость доменной модели от конкретного провайдера или внешнего AI/RAG-движка.

### ImpactMap

Структурированная карта влияния.

Поля:

- changeSummary;
- affectedRequirements;
- affectedTasks;
- affectedProjectDecisions;
- affectedApiInterfacesDocumentsTests;
- affectedArchitecturalConstraints;
- affectedOrganizationalContextItems;
- contradictions;
- missingInformation;
- clarificationQuestions;
- risks;
- optionsForExpertReview;
- preliminaryAssessment.

Каждый элемент карты влияния должен иметь стабильный идентификатор внутри анализа, чтобы экспертная оценка и JSON-экспорт могли ссылаться на него воспроизводимо.

Каждый элемент карты влияния имеет:

- id;
- title;
- description;
- severity или importance, если применимо;
- relatedContextFragmentIds, если связь доступна;
- notes.

Связь с фрагментами контекста является полезной, но не обязательной для каждого элемента.

### ExpertEvaluation

Экспертная проверка аналитического результата.

Поля:

- id;
- analysisId;
- contextSufficiency;
- resultUsefulness;
- generalComment;
- evaluatedItems;
- missedItems;
- corrections.

Оценка элемента:

- targetType;
- targetId;
- mark;
- comment;
- correctionText.

`targetType` и `targetId` должны ссылаться на стабильный элемент или группу элементов карты влияния. Если эксперт фиксирует пропущенный элемент, который отсутствовал в результате интеллектуального анализа, для него создается отдельная запись в `missedItems`.

Значения `mark`:

- confirmed;
- corrected;
- rejected;
- needsClarification.

### ExpertConclusion

Итоговая фиксация экспертного заключения.

Поля:

- id;
- analysisId;
- conclusionType;
- comment;
- rationale;
- fixedAt.

Типы заключения:

- accept;
- acceptWithLimitations;
- sendForClarification;
- splitIntoSeveralTasks;
- reject;
- returnForReanalysis.

Эти типы являются только фиксируемыми выводами эксперта. Они не запускают workflow, не создают задачи и не выполняют действий во внешних системах.

## Состояния анализа

Статус описывает состояние артефакта анализа, а не маршрут согласования.

Предлагаемые статусы:

- draft;
- inputIncomplete;
- readyForAnalysis;
- llmAnalysisRunning;
- llmAnalysisCompleted;
- llmAnalysisFailed;
- needsExpertEvaluation;
- returnedForClarification;
- needsReanalysis;
- expertConclusionFixed;
- exported.

В UI эти статусы могут быть сгруппированы проще. В хранилище они нужны для воспроизводимого описания жизненного состояния артефакта.

Статусы являются пассивными метками состояния. Они не запускают маршруты согласования, не создают задачи, не отправляют уведомления и не выполняют действия во внешних системах.

## Контракт интеллектуального анализа

`IAiAnalysisEngine` должен возвращать структурированный аналитический результат, а не свободный чат-ответ и не управленческое решение.

Вход интеллектуального анализа:

- исходное описание или требование;
- проектный запрос, предлагаемое изменение или уточнение;
- описание ситуации;
- источник запроса или изменения;
- список фрагментов связанного контекста, если добавлены;
- структура ожидаемого результата;
- указание, что итоговое решение принимает человек.

Ожидаемый выход:

- краткое описание изменения;
- потенциально затронутые требования;
- потенциально затронутые задачи;
- потенциально затронутые проектные решения;
- потенциально затронутые API, интерфейсы, документы и тесты;
- потенциально затронутые архитектурные ограничения;
- потенциально затронутые элементы организационного контекста;
- противоречия;
- недостающая информация;
- вопросы для уточнения;
- риски;
- варианты для экспертного рассмотрения;
- предварительная аналитическая оценка;
- сведения о неопределенности, особенно если контекст отсутствует или неполон.

В технической реализации контракт интеллектуального анализа должен быть представлен в виде внутренней структуры данных. Конкретный prompt, формат response schema и механизм валидации относятся к implementation-plan.

До начала кодирования implementation-plan должен определить минимально валидную структуру результата, поведение при частично корректном ответе, поведение при неструктурированном ответе и способ сохранения raw response как диагностического fallback.

## Границы AI/RAG-компонентов

В техническом проекте явно разделяются:

- собственная программа: хранит анализы, контекст, карту влияния, экспертную оценку, экспертное заключение и экспорт;
- внешний LLM-провайдер: принимает подготовленный prompt и возвращает raw response или структурированный ответ;
- внешний AI/RAG-движок: при будущей реализации может искать и подготавливать релевантный контекст через адаптер `IAiAnalysisEngine`;
- vector store: если появится позже, остается инфраструктурной частью внешнего AI/RAG-движка или отдельной инфраструктурой, а не ядром собственной программы.

Dify, Flowise, LangGraph, Haystack или аналогичные системы рассматриваются как внешние adapter-компоненты. Собственная программа не должна зависеть от конкретного внешнего AI/RAG-движка.

## Хранение данных

SQLite используется как основное хранилище.

В SQLite хранятся:

- анализы;
- фрагменты контекста;
- результаты интеллектуального анализа;
- карты влияния;
- экспертные оценки;
- экспертные заключения;
- метаданные загруженных файлов;
- метаданные экспорта.

Загруженные файлы хранятся в локальном каталоге приложения, например:

```text
data/uploads/{analysisId}/{fileName}
```

В базе сохраняются путь, исходное имя файла, тип, дата загрузки и извлеченный текст.

Для MVP не требуется:

- отдельное файловое хранилище;
- объектное хранилище;
- индекс полнотекстового поиска;
- vector database;
- база знаний по всем анализам.

## Экспорт

### Markdown

Markdown-отчет предназначен для чтения человеком, вставки в проектные материалы или использования в диссертационной работе.

Состав:

- название анализа;
- дата и статус;
- исходное требование;
- предлагаемое изменение;
- описание ситуации и источник;
- список контекстных фрагментов;
- карта влияния;
- риски, противоречия, вопросы и недостающая информация;
- экспертная оценка;
- экспертное заключение, комментарий и основание;
- пояснение, что LLM сформировала предварительный аналитический материал, а не управленческое решение.

### JSON

JSON-экспорт предназначен для повторной обработки и экспериментальной проверки.

Состав:

- metadata;
- input;
- contextFragments;
- aiAnalysisResult;
- impactMap;
- expertEvaluation;
- expertConclusion;
- exportMetadata.

JSON Schema на этапе технического проекта не фиксируется как отдельный стандарт, но при реализации имена полей должны быть стабильными, чтобы результаты можно было использовать в главе 4.

Для главы 4 особенно важны стабильные идентификаторы элементов карты влияния, типы элементов, экспертные отметки, пропущенные элементы, корректировки и основание экспертного заключения. Эти поля должны сохраняться в JSON-экспорте в предсказуемом виде после начала реализации.

## UI-структура MVP

UI реализуется как набор страниц Razor Pages.

Минимальный пользовательский путь:

1. Список сохраненных анализов.
2. Создание или редактирование анализа.
3. Добавление связанного контекста.
4. Проверка входных данных и запуск интеллектуального анализа.
5. Просмотр карты влияния.
6. Экспертная оценка и фиксация экспертного заключения.
7. Экспорт результата.

UI не должен быть чат-интерфейсом. Чат может быть внутренней метафором взаимодействия с LLM, но пользовательский результат должен быть представлен как структурированная карта влияния и экспертная фиксация.

UI не должен включать:

- доски задач;
- backlog;
- спринты;
- назначение исполнителей;
- маршруты согласования;
- автоматические действия во внешних системах.

## Интеграция с LLM-провайдером

В приложении вводится интерфейс LLM-провайдера как нижний provider boundary для `DirectLlmAnalysisEngine`.

Ответственность интерфейса:

- принять подготовленный запрос;
- вернуть raw response;
- вернуть структурированный результат или ошибку;
- сохранить минимальные метаданные выполнения.

Первая реализация может работать с одним выбранным провайдером через конфигурацию приложения. Ключи доступа и настройки провайдера не хранятся в репозитории.

Доменная логика, UI и application services не должны зависеть от конкретного провайдера, модели или SDK. Они работают с результатом `IAiAnalysisEngine`; прямой LLM-вызов остается внутри реализации analysis engine и provider adapter.

## Ошибки и повторный анализ

Если интеллектуальный анализ завершился ошибкой, приложение сохраняет состояние ошибки и текст ошибки, пригодный для диагностики.

Повторный анализ в MVP допускается как запуск нового интеллектуального анализа для того же артефакта после изменения входных данных или контекста.

Для MVP можно хранить только последний рабочий результат и последний raw response до момента экспертной фиксации. После фиксации экспертной оценки или экспорта результат, на основании которого выполнялась оценка, должен оставаться воспроизводимым и не должен быть незаметно перезаписан повторным анализом.

Если в ходе implementation planning будет принято решение поддержать несколько запусков анализа для одного артефакта, история запусков должна быть ограниченной и не должна превращать MVP в систему аудита LLM-экспериментов.

## Данные для главы 4

Для экспериментальной проверки важно сохранить:

- исходные данные анализа;
- состав контекста;
- карту влияния;
- экспертные отметки;
- пропущенные элементы;
- корректировки;
- экспертное заключение;
- основание заключения;
- дату фиксации;
- JSON-экспорт.

Сбор времени выполнения анализа и сравнение трудоемкости могут выполняться методикой эксперимента вне программы. В MVP можно не реализовывать таймеры и специальные экспериментальные панели.

## Безопасность и конфиденциальность

MVP предназначен для локальной демонстрации и работы с подготовленными или обезличенными материалами.

Минимальные решения:

- не хранить ключи LLM-провайдера в репозитории;
- использовать конфигурацию окружения для секретов;
- явно предупреждать, что пользователь сам выбирает материалы, передаваемые в LLM;
- не подключать реальные корпоративные источники в MVP.

Авторизация пользователей не входит в MVP, если приложение используется как локальный демонстрационный прототип. Если потребуется демонстрация нескольким пользователям, вопрос авторизации должен быть решен отдельно до реализации.

## Что откладывается

На следующих этапах не включается в первый MVP:

- реальные интеграции с Jira, GitLab, Confluence, почтой, чатами;
- собственный RAG, embeddings, rerank и vector database;
- интеграция с Dify, Flowise, LangGraph, Haystack или аналогичным AI/RAG-движком;
- поиск по всем сохраненным анализам как база знаний;
- многоуровневая ролевая модель;
- workflow согласования;
- назначение исполнителей;
- автоматическое изменение требований, документов, API или кода;
- автоматическое создание pull request;
- PDF-экспорт;
- dashboards для исследования;
- микросервисная архитектура.

## Риски технического проекта

Основные риски:

- LLM может вернуть неполный или плохо структурированный результат;
- слишком детальная экспертная разметка может усложнить MVP;
- отсутствие связи каждого вывода с фрагментом контекста может снизить трассируемость;
- cloud LLM может быть несовместим с конфиденциальными данными;
- повторные запуски LLM могут потребовать истории версий;
- JSON-экспорт должен быть достаточно стабильным для главы 4.

Снижение рисков:

- сохранять raw response и структурированный результат;
- явно показывать неопределенность и неполноту контекста;
- делать связь с контекстом опциональной;
- ограничить экспертную разметку элементами или группами элементов;
- использовать подготовленные демонстрационные данные;
- стабилизировать JSON-поля в implementation-plan.

## Следующий шаг

После утверждения технического проекта можно переходить к implementation planning:

- определить структуру репозитория и проектов;
- описать первые task реализации;
- выбрать минимальный набор тестов;
- подготовить план первого проверяемого diff;
- только после этого переходить к написанию кода по отдельному явному решению.
