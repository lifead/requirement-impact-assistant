# Уточнения и зафиксированные решения для SDD MVP-1

## Назначение документа

Документ фиксирует уточнения, рамочные решения и открытые вопросы для SDD MVP-1 программы анализа влияния проектных запросов.

Документ является промежуточным артефактом между стратегией MVP-1 и будущим `requirements-draft.md`. Он не является требованиями, техническим проектом, implementation plan, task breakdown или описанием конкретной реализации.

Назначение документа - снять ключевую неопределенность перед подготовкой требований MVP-1, зафиксировать границы развития после MVP-0 и отделить принятые решения от вопросов, которые еще нужно закрыть до перехода к requirements-draft.

## Основание

Основанием для документа являются:

- зафиксированная база MVP-0 в `docs/program/mvp0/`;
- стратегический план развития `docs/program/mvp1/mvp1-strategy.md`;
- решение развивать MVP-1 как расширение существующей программы, а не как новую систему;
- решение добавить внешний AI/RAG-контур за application-level границей `IAiAnalysisEngine`.

MVP-0 считается реализованной и зафиксированной базой. Документы MVP-0 в рамках MVP-1 не переоткрываются и не переписываются. MVP-1 должен опираться на них как на исходную точку развития.

## Принятые рамочные решения

MVP-1 расширяет MVP-0, а не заменяет его.

В MVP-1 сохраняются основные возможности MVP-0: карточка анализа, ручной контекст, запуск интеллектуального анализа, сохранение результата, экспертная оценка, экспертное заключение и экспорт.

`IAiAnalysisEngine` остается главной application-level границей для всех интеллектуальных сценариев. UI, страницы, controllers, page models и бизнес-логика не должны вызывать внешний AI/RAG-движок, Dify, retrieval API или LLM напрямую. Вызов интеллектуального анализа должен проходить через application-level границу `IAiAnalysisEngine` и согласованные application services.

`DirectLlmAnalysisEngine` сохраняется как отдельный режим direct LLM analysis. Он не удаляется и не считается устаревшим только из-за появления внешнего AI/RAG-режима.

Новый режим анализа должен проходить через `ExternalRagAnalysisEngine` или аналогичный adapter, реализующий ту же application-level границу. Название может быть уточнено на этапе проектирования, но смысл решения фиксируется уже сейчас: внешний AI/RAG-контур подключается как сменяемый adapter, а не как новая архитектурная основа приложения.

Dify может быть первой минимальной реализацией такого adapter/provider для проверки подхода. При этом Dify не становится обязательной основой проекта, не определяет всю архитектуру программы и не должен быть единственным возможным внешним AI/RAG-движком.

## Роль внешнего AI/RAG-движка

Внешний AI/RAG-движок отвечает за retrieval, подбор контекста и подготовку retrieval-assisted аналитического результата.

Собственная программа отвечает за:

- карточку анализа;
- сохранение исходных данных;
- сохранение ручного контекста;
- вызов выбранного режима анализа через `IAiAnalysisEngine`;
- сохранение результата интеллектуального анализа;
- сохранение retrieved context или metadata, если они доступны;
- экспертную оценку человеком;
- экспертное заключение;
- экспорт;
- воспроизводимое представление сохраненного анализа.

MVP-1 не должен проектировать собственный retrieval pipeline. В рамках MVP-1 не вводятся собственные embeddings, vector database, rerank, agentic search или собственная RAG-платформа.

## Режимы анализа

В MVP-1 должны различаться как минимум два режима анализа:

- direct LLM analysis;
- external RAG analysis.

Direct LLM analysis использует проектный запрос и manual context, подготовленный пользователем внутри программы.

External RAG analysis использует проектный запрос и может обращаться к внешнему AI/RAG-движку, который сам подбирает контекст. При этом программа может передавать во внешний контур исходные данные анализа и, если это будет признано допустимым в требованиях, manual context как часть входных материалов.

Оба режима должны оставаться сопоставимыми по смыслу результата. Это нужно для экспертной оценки, экспорта и будущей экспериментальной проверки.

## Manual context и retrieved context

В MVP-1 нужно явно различать manual context и retrieved context.

Manual context - это контекст, который пользователь добавил в программу вручную или через подготовленные файлы. Этот контекст уже входит в логику MVP-0 и остается частью карточки анализа.

Retrieved context - это контекст, который был найден или выбран внешним AI/RAG-движком в ходе external RAG analysis. Он не должен смешиваться с manual context так, будто был введен пользователем.

Если внешний AI/RAG-движок возвращает retrieved context, программа должна сохранить его или доступные metadata отдельно от manual context.

Если внешний AI/RAG-движок возвращает только metadata без полного текста фрагментов, программа должна сохранить эти metadata и явно зафиксировать ограничение.

Если внешний AI/RAG-движок не возвращает retrieved context и не возвращает пригодные metadata, программа должна явно фиксировать, что основание внешнего retrieval недоступно для воспроизводимого просмотра внутри программы.

## Минимальное основание результата

Для MVP-1 важно сохранить основание результата настолько полно, насколько это позволяет внешний движок.

Если эти сведения доступны от внешнего движка, для будущих требований целесообразно предусмотреть их сохранение:

- текст или excerpt найденного фрагмента;
- название источника;
- идентификатор источника или фрагмента;
- ссылку или внешний reference, если он есть;
- rank, score или аналогичную оценку релевантности;
- сведения о внешнем движке или adapter, который вернул контекст;
- сведения о том, что retrieved context был недоступен или неполон.

На этом этапе не фиксируется конкретная структура хранения, схема базы данных или JSON schema. Фиксируется только смысловое решение: external RAG результат должен по возможности иметь проверяемое основание, а отсутствие такого основания должно быть видно.

## Сопоставимость direct LLM и external RAG

MVP-1 должен сохранять сопоставимость direct LLM и external RAG режимов.

Сопоставимость нужна не для того, чтобы оба режима давали одинаковый результат, а для того, чтобы в будущем можно было сравнить:

- полноту выявленных затронутых элементов;
- качество вопросов для уточнения;
- качество выявленных рисков;
- наличие или отсутствие противоречий;
- полезность результата для эксперта;
- степень подтверждения или корректировки выводов экспертом;
- влияние retrieved context на качество анализа.

Поэтому результат external RAG режима должен приводиться к той же общей логике карты влияния, экспертной оценки и экспорта, которая уже используется в MVP-0.

## Экспертное решение

Экспертное решение остается отдельно от результата AI/RAG/LLM.

AI/RAG/LLM формирует предварительный аналитический материал. Он может помочь выявить затронутые элементы, риски, противоречия, недостающую информацию и вопросы для уточнения, но не принимает управленческое решение.

Человек проверяет результат, оценивает его применимость и фиксирует экспертное заключение.

Программа не утверждает изменение, не отклоняет его, не назначает исполнителей, не создает задачи, не инициирует изменения требований, документов, API или кода.

## Роль Dify

Dify рассматривается как возможная первая минимальная реализация adapter/provider для MVP-1.

Цель использования Dify - проверить практическую жизнеспособность external RAG режима и показать, что программа может работать с внешним AI/RAG-контуром через согласованную границу.

Dify не должен становиться обязательной технологической основой проекта. Требования MVP-1 не должны формулироваться так, будто программа является оболочкой над Dify.

Если в дальнейшем будет выбран другой внешний AI/RAG-движок, общая модель MVP-1 должна остаться применимой.

## Границы MVP-1

В MVP-1 входит:

- развитие существующего контура анализа MVP-0;
- сохранение `IAiAnalysisEngine` как главной границы интеллектуального анализа;
- сохранение direct LLM режима;
- добавление external RAG режима через adapter;
- минимальная практическая проверка подхода через Dify adapter/provider или аналогичный внешний adapter;
- различение manual context и retrieved context;
- сохранение retrieved context или metadata, если внешний движок их возвращает;
- явная фиксация ограничения, если retrieved context недоступен;
- сохранение сведений о выбранном режиме анализа;
- расширение результата и экспорта сведениями, нужными для сопоставления режимов;
- сохранение экспертной оценки и экспертного решения как отдельного человеческого слоя.

В MVP-1 не входит:

- собственная RAG-платформа;
- собственные embeddings;
- собственная vector database;
- собственный rerank;
- agentic workflow или многошаговый агентный поиск;
- обязательная зависимость от Dify;
- интеграции с Jira, Confluence, GitLab, ALM, почтой, чатами или корпоративными workflow-системами;
- управление задачами, backlog, спринтами, назначениями и согласованиями;
- автоматическое изменение требований, документов, API или кода;
- замена экспертного заключения автоматическим решением AI/RAG/LLM.

## Решения перед requirements-draft

Перед подготовкой requirements-draft для MVP-1 предварительно приняты следующие решения.

MVP-1 должен быть сформулирован как развитие MVP-0 через второй режим интеллектуального анализа.

Требования должны описывать поведение программы на уровне сценариев, данных результата и воспроизводимости, но не должны преждевременно фиксировать структуру таблиц, endpoint-ы, конкретную JSON schema, UI-формы или внутренние классы.

Требования должны явно учитывать два режима анализа: direct LLM и external RAG.

Требования должны предусмотреть сохранение сведений о режиме анализа, чтобы в сохраненном анализе и экспорте было видно, каким способом получен результат.

Требования должны зафиксировать, что для external AI/RAG provider полный raw response не сохраняется по умолчанию. По умолчанию сохраняются только sanitized diagnostic snapshot, metadata response или иные ограниченные диагностические сведения без секретов, токенов, персональных данных и чувствительных материалов. Полный raw response допустим только по отдельному явному решению или в debug/manual режиме и не становится стабильной публичной моделью программы.

Требования должны предусмотреть отдельное представление manual context и retrieved context.

Требования должны предусмотреть поведение для случая, когда внешний AI/RAG-движок вернул результат, но не вернул retrieved context.

Требования должны сохранить границу ответственности: внешний движок подбирает контекст, а программа сохраняет анализ, результат, экспертную оценку, экспорт и ограничения воспроизводимости.

## Что пока не фиксируется

На этапе clarification-decisions для MVP-1 пока не фиксируются:

- окончательные функциональные требования;
- окончательные нефункциональные требования;
- технический проект;
- implementation plan;
- task breakdown;
- структура базы данных;
- объектная модель;
- API-контракты;
- endpoint-ы;
- конкретные UI-формы;
- конкретный формат response внешнего AI/RAG-движка;
- конкретная JSON schema;
- конкретный формат prompt или system instructions;
- технические детали Dify adapter;
- способ авторизации во внешнем AI/RAG-движке;
- способ хранения секретов;
- тестовые сценарии;
- методика экспериментального сравнения режимов.

Эти вопросы должны рассматриваться на последующих этапах workflow после утверждения требований или в рамках отдельного исследовательского решения, если оно потребуется.

12. Нужно ли в MVP-1 хранить несколько запусков анализа для одного артефакта, если пользователь сравнивает direct LLM и external RAG режимы, или достаточно хранить выбранные результаты в ограниченном виде?

## Критерий готовности к requirements-draft

Переход к requirements-draft MVP-1 возможен, когда будут достаточно зафиксированы:

- границы MVP-1 как расширения MVP-0;
- роль `IAiAnalysisEngine` как главной application-level границы;
- различие direct LLM и external RAG режимов;
- роль `ExternalRagAnalysisEngine` или аналогичного adapter;
- необязательная, проверочная роль Dify;
- различие manual context и retrieved context;
- правила фиксации retrieved context или ограничения его отсутствия;
- принцип сопоставимости режимов для будущей экспериментальной проверки;
- отделение экспертного решения от результата AI/RAG/LLM;
- перечень вопросов, которые можно безопасно перенести в техническое проектирование.
