# Черновик требований SDD MVP-1

## Назначение документа

Документ фиксирует содержательный черновик требований к SDD MVP-1 программы анализа влияния проектных запросов.

Документ является этапом requirements-draft. Он не является техническим проектом, описанием архитектуры, implementation plan, task breakdown, набором пользовательских историй или выбором технологического стека.

Назначение документа - зафиксировать требования к поведению MVP-1 на уровне пользовательских сценариев, результата анализа, основания результата, экспертной оценки, сохранения и экспорта без преждевременной фиксации технических решений.

MVP-1 рассматривается как развитие MVP-0. Он не заменяет существующий контур direct LLM analysis, а добавляет второй режим интеллектуального анализа через внешний AI/RAG-контур с сохранением воспроизводимого основания результата там, где это возможно.

## Основание

Основанием для требований являются:

- `docs/program/mvp1/mvp1-strategy.md`;
- `docs/program/mvp1/clarification-decisions.md`;
- зафиксированные материалы MVP-0 в `docs/program/mvp0/`;
- существующая позиция проекта о том, что AI/RAG/LLM формирует предварительный аналитический материал, а экспертное заключение и управленческое рассмотрение остаются за человеком.

Из этих документов в требования перенесены следующие рамочные решения:

- MVP-1 расширяет MVP-0, а не переписывает его;
- direct LLM analysis сохраняется как отдельный режим анализа;
- external AI/RAG analysis добавляется как второй режим анализа;
- все интеллектуальные сценарии должны запускаться через application-level границу `IAiAnalysisEngine`;
- внешний AI/RAG-движок отвечает за retrieval, подбор контекста и подготовку retrieval-assisted аналитического результата;
- программа отвечает за карточку анализа, сохранение исходных данных, результата, retrieved context или metadata, экспертной оценки, экспертного заключения и экспорта;
- manual context и retrieved context должны различаться;
- отсутствие retrieved context или неполнота metadata должны явно отображаться пользователю;
- результат direct LLM и результат external AI/RAG режимов должны оставаться сопоставимыми для экспертной оценки, экспорта и будущей экспериментальной проверки;
- программа не должна превращаться в RAG-платформу, workflow-систему, dashboard или оболочку над одним внешним инструментом.

## Назначение MVP-1

MVP-1 должен поддержать анализ влияния проектного запроса в двух режимах:

- direct LLM;
- external AI/RAG.

Direct LLM режим использует исходные данные анализа и manual context, подготовленный пользователем внутри программы.

External AI/RAG режим использует исходные данные анализа и обращается к внешнему AI/RAG-движку, который может подобрать retrieved context и вернуть retrieval-assisted аналитический результат.

Оба режима должны формировать предварительный аналитический материал для человека. Программа не должна утверждать изменение, отклонять изменение, назначать исполнителей, создавать задачи, изменять требования, документы, API или код.

Результатом MVP-1 должен быть сохраненный и экспортируемый артефакт анализа, в котором видно:

- какой режим анализа был использован;
- каким engine/provider был получен результат;
- какие исходные данные и manual context участвовали в анализе;
- какой retrieved context или metadata были доступны для external AI/RAG режима;
- какие ограничения воспроизводимости есть у результата;
- какая экспертная оценка и какое экспертное заключение были зафиксированы человеком.

## Границы MVP-1

В MVP-1 входит:

- сохранение возможностей MVP-0 по созданию карточки анализа, manual context, запуску анализа, сохранению результата, экспертной оценке, экспертному заключению и экспорту;
- сохранение пригодности ранее созданных результатов MVP-0 для просмотра и экспорта;
- выбор режима анализа перед запуском: direct LLM или external AI/RAG;
- запуск выбранного режима интеллектуального анализа через `IAiAnalysisEngine`;
- получение структурированного результата от direct LLM или external AI/RAG engine;
- сохранение сведений о режиме анализа, использованном analysis engine и provider/adapter, если он применялся;
- сохранение retrieved context или metadata, если внешний AI/RAG-движок их возвращает;
- отдельное представление manual context и retrieved context;
- отображение пользователю основания результата external AI/RAG режима;
- явное отображение ограничения, если retrieved context недоступен или неполон;
- расширение Markdown и JSON export сведениями о режиме анализа, engine/provider и основании результата;
- сохранение экспертной оценки и экспертного заключения отдельно от результата AI/RAG/LLM;
- обеспечение сопоставимости результатов direct LLM и external AI/RAG режимов.

В MVP-1 не входит:

- собственная RAG-платформа;
- собственные embeddings;
- собственная vector database;
- собственный rerank;
- agentic search или многошаговый agentic workflow;
- интеграции с Jira, Confluence, GitLab, ALM, почтой, чатами или корпоративными workflow-системами;
- dashboard, backlog, управление задачами, назначениями, согласованиями или спринтами;
- автоматическое изменение требований, документов, API или кода;
- обязательная генерация PDF;
- замена экспертного заключения автоматическим решением AI/RAG/LLM;
- фиксация конкретной структуры базы данных, endpoint-ов, JSON schema, UI-форм, prompt-ов или response-формата внешнего движка.

## Пользовательские роли и сценарии

На уровне требований выделяются смысловые роли, а не техническая ролевая модель доступа.

**Пользователь анализа** создает или открывает карточку анализа, вводит исходные данные проектного запроса, добавляет manual context, выбирает режим анализа, запускает интеллектуальный анализ и просматривает результат.

**Эксперт** просматривает результат интеллектуального анализа, основание результата, ограничения воспроизводимости, оценивает применимость результата и фиксирует экспертное заключение.

**Исследователь-экспериментатор** использует сохраненные анализы и экспортированные результаты для дальнейшего сравнения direct LLM и external AI/RAG режимов.

Основной пользовательский сценарий MVP-1:

1. Пользователь создает или открывает анализ проектного запроса.
2. Пользователь заполняет исходные данные анализа.
3. Пользователь добавляет manual context при необходимости.
4. Пользователь выбирает режим анализа: direct LLM или external AI/RAG.
5. Программа показывает, какой режим будет использован, и какие исходные материалы доступны для запуска.
6. Пользователь запускает анализ.
7. Программа выполняет анализ через `IAiAnalysisEngine`.
8. Программа сохраняет результат анализа, сведения о режиме, engine/provider и доступное основание результата.
9. Пользователь просматривает структурированную карту влияния и основание результата.
10. Эксперт фиксирует экспертную оценку и экспертное заключение отдельно от AI/RAG/LLM результата.
11. Пользователь экспортирует результат в Markdown или JSON.

## Функциональные требования

FR-1. Программа должна сохранять существующий сценарий MVP-0 по созданию анализа проектного запроса, добавлению manual context, запуску анализа, сохранению результата, экспертной оценке, экспертному заключению и экспорту.

FR-2. Существующие сохраненные анализы MVP-0 должны оставаться пригодными для просмотра и экспорта после перехода к MVP-1. Если в сохраненном анализе MVP-0 отсутствуют сведения о режиме анализа, engine или provider, программа должна отображать это как отсутствие сведений у ранее созданного результата, а не как ошибку анализа.

FR-3. Программа должна позволять пользователю выбрать режим интеллектуального анализа перед запуском.

FR-4. Минимальный набор режимов анализа должен включать direct LLM и external AI/RAG.

FR-5. Программа должна явно показывать выбранный режим анализа до запуска и после получения результата.

FR-6. Запуск интеллектуального анализа в любом режиме должен выполняться через `IAiAnalysisEngine`.

FR-7. UI, страницы, controllers, page models и бизнес-логика не должны рассматриваться как самостоятельные инициаторы прямого обращения к LLM, retrieval API или внешнему AI/RAG-движку; пользовательский сценарий должен проходить через application-level boundary `IAiAnalysisEngine`.

FR-8. В direct LLM режиме программа должна использовать исходные данные анализа и manual context, если он был добавлен пользователем.

FR-9. В external AI/RAG режиме программа должна передавать исходные данные анализа во внешний AI/RAG-контур через выбранный analysis engine.

FR-10. В external AI/RAG режиме manual context может передаваться во внешний AI/RAG-контур только как отдельная часть входных материалов, если пользователь явно выбрал его использование для данного запуска анализа. Независимо от передачи во внешний контур, manual context должен сохраняться и отображаться отдельно от retrieved context.

FR-11. Программа должна получать от external AI/RAG engine структурированный аналитический результат, пригодный для представления в общей логике карты влияния.

FR-12. Программа должна сохранять результат external AI/RAG анализа даже в случае, если внешний движок вернул аналитический результат без полного retrieved context, при условии что сам результат пригоден для просмотра.

FR-13. Если внешний AI/RAG-движок вернул retrieved context, программа должна сохранить его отдельно от manual context.

FR-14. Если внешний AI/RAG-движок вернул metadata вместо полного текста retrieved context, программа должна сохранить доступные metadata отдельно от manual context.

FR-15. Если внешний AI/RAG-движок не вернул retrieved context и не вернул пригодные metadata, программа должна явно зафиксировать, что основание внешнего retrieval недоступно для воспроизводимого просмотра внутри программы.

FR-16. Программа должна показывать пользователю основание результата external AI/RAG режима: retrieved context, metadata или ограничение отсутствия такого основания.

FR-17. Программа должна явно отличать ограничения результата от самого результата анализа. Ограничение отсутствия retrieved context не должно выглядеть как экспертное заключение или как ошибка эксперта.

FR-18. Программа должна сохранять для каждого результата интеллектуального анализа сведения о режиме анализа, использованном analysis engine и внешнем provider/adapter, если он применялся. Конкретные имена полей и структура хранения определяются на этапе technical design.

FR-18a. Для external AI/RAG режима программа не должна по умолчанию сохранять полный raw response внешнего provider/adapter. По умолчанию должны сохраняться только sanitized diagnostic snapshot, metadata response или иные ограниченные диагностические сведения, достаточные для диагностики и воспроизводимости без раскрытия секретов, токенов, персональных данных и чувствительных материалов.

FR-18b. Полный raw response внешнего provider/adapter может сохраняться только по отдельному явному решению или в debug/manual режиме. Такой raw response не является стабильной публичной моделью программы, а Dify-specific response не должен попадать в UI, domain model или стабильный JSON export как обязательная структура.

FR-19. Программа должна показывать пользователю сведения о режиме анализа и, на уровне, достаточном для понимания результата, сведения об engine/provider.

FR-20. Программа должна сохранять direct LLM и external AI/RAG результаты в сопоставимой смысловой структуре, пригодной для экспертной оценки и экспорта.

FR-21. Программа должна позволять эксперту оценивать результат анализа независимо от выбранного режима.

FR-22. Программа должна позволять эксперту зафиксировать экспертное заключение отдельно от результата AI/RAG/LLM.

FR-23. Программа не должна смешивать экспертную оценку, экспертное заключение и комментарий человека с AI/RAG/LLM результатом так, будто они имеют один источник.

FR-24. Программа должна позволять экспортировать результат анализа в Markdown с учетом сведений MVP-1.

FR-25. Программа должна позволять экспортировать результат анализа в JSON с учетом сведений MVP-1.

FR-26. Программа должна сохранять результат анализа так, чтобы при последующем просмотре было понятно, каким режимом он был получен и какие ограничения есть у его основания.

## Требования к данным результата и экспорту

Сохраненный результат MVP-1 должен включать данные MVP-0 и расширяться сведениями, необходимыми для режима external AI/RAG.

Сохраненный результат должен включать:

- исходные данные анализа;
- manual context, если он был добавлен;
- выбранный режим анализа;
- сведения об использованном analysis engine;
- сведения о provider/adapter, если он применялся;
- результат интеллектуального анализа;
- структурированную карту влияния;
- retrieved context или metadata, если они доступны;
- признак или описание ограничения, если retrieved context недоступен или неполон;
- экспертную оценку;
- экспертное заключение;
- комментарий и основание экспертного заключения;
- дату фиксации результата или изменения статуса;
- статус анализа.

Для retrieved context или metadata программа должна сохранять доступные сведения на смысловом уровне, если они возвращены внешним движком:

- текст фрагмента или excerpt;
- название источника;
- идентификатор источника или фрагмента;
- ссылка или внешний reference;
- rank, score или аналогичная оценка релевантности;
- сведения о provider/engine, который вернул контекст;
- признак неполноты или недоступности retrieved context.

Конкретная структура хранения, набор таблиц, объектная модель и JSON schema на этапе требований не фиксируются.

Markdown export должен быть человекочитаемым и включать:

- исходные данные анализа;
- manual context или сведения о его отсутствии;
- режим анализа;
- сведения об использованном analysis engine;
- сведения о provider/adapter, если он применялся;
- структурированную карту влияния;
- retrieved context или metadata для external AI/RAG режима, если они доступны;
- явное ограничение, если retrieved context недоступен или неполон;
- экспертную оценку;
- экспертное заключение;
- комментарий и основание экспертного заключения;
- дату и статус.

JSON export должен быть пригоден для последующего анализа и сравнения режимов. Он должен включать:

- исходные данные анализа;
- сведения о manual context;
- режим анализа;
- сведения об использованном analysis engine;
- сведения о provider/adapter, если он применялся;
- результат интеллектуального анализа;
- структурированную карту влияния;
- сведения о retrieved context или metadata;
- сведения об ограничениях retrieved context;
- экспертную оценку;
- экспертное заключение;
- комментарий эксперта;
- основание экспертного заключения;
- дату;
- статус.

JSON export должен позволять в дальнейшем сравнивать direct LLM и external AI/RAG результаты по общим смысловым полям, не требуя одинаковости внутренних механизмов получения результата.

## Требования к сопоставимости режимов

MVP-1 должен обеспечивать сопоставимость результатов direct LLM и external AI/RAG режимов.

Сопоставимость означает, что результаты разных режимов можно рассматривать в общей логике:

- затронутые элементы;
- риски;
- противоречия;
- недостающая информация;
- вопросы для уточнения;
- варианты для экспертного рассмотрения;
- предварительная аналитическая оценка;
- экспертная оценка;
- экспертное заключение.

Сопоставимость не означает, что оба режима должны давать одинаковые выводы.

Сопоставимость нужна для того, чтобы в дальнейшем можно было сравнивать:

- полноту выявленных затронутых элементов;
- качество вопросов для уточнения;
- качество выявленных рисков;
- наличие противоречий;
- полезность результата для эксперта;
- степень подтверждения, корректировки или отклонения выводов экспертом;
- влияние retrieved context на качество анализа.

Программа должна сохранять различие источников результата: direct LLM результат не должен искусственно иметь retrieved context, а external AI/RAG результат должен показывать retrieved context или ограничение его отсутствия.

## Ограничения

AI/RAG/LLM результат является предварительным аналитическим материалом и может быть неполным, ошибочным или спорным.

Retrieved context, возвращенный внешним AI/RAG-движком, не гарантирует полноту всей корпоративной базы знаний и не должен представляться как исчерпывающее основание управленческого решения.

Retrieved context и metadata являются основанием предварительного AI/RAG анализа, но не заменяют экспертное основание итогового заключения. Эксперт вправе подтвердить, скорректировать или отклонить выводы AI/RAG/LLM независимо от retrieved context.

Если retrieved context недоступен, программа должна явно показывать это как ограничение воспроизводимости результата.

Если retrieved context доступен только частично, программа должна показывать, что основание результата неполно.

Программа не должна скрывать неопределенность: вопросы, недостающая информация, спорные выводы, ограничения внешнего retrieval и признаки неполноты должны быть видимы пользователю.

Использование внешнего AI/RAG-движка не должно превращать программу в субъект управления. Экспертное рассмотрение и итоговое заключение остаются за человеком.

Конкретный внешний provider может использоваться для проверки подхода, но требования MVP-1 не должны делать программу зависимой от одного продукта или описывать ее как оболочку над одним внешним инструментом.

## Non-goals

MVP-1 не должен включать:

- собственную реализацию embeddings;
- собственную vector database;
- собственный rerank;

## Открытые вопросы и критерий готовности к technical design

Следующие вопросы можно перенести в technical design без изменения смысла требований:

- как именно представить выбор режима анализа в пользовательском интерфейсе;
- как технически хранить сведения о режиме анализа, analysis engine и provider/adapter;
- как технически представить retrieved context и metadata без фиксации лишней сложности;
- как обработать разные уровни полноты ответа внешнего AI/RAG-движка;
- как технически связать retrieved context с элементами карты влияния, если внешний движок возвращает такие связи;
- как обеспечить экспорт Markdown и JSON без преждевременной фиксации конкретной JSON schema;
- как технически реализовать provider/adapter для первого внешнего AI/RAG-движка;
- как хранить секреты и настройки внешнего provider;
- какие тестовые сценарии нужны для проверки direct LLM и external AI/RAG режимов.

К technical design можно переходить, когда утверждены следующие положения требований:

- MVP-1 расширяет MVP-0, а не заменяет его;
- direct LLM и external AI/RAG являются двумя поддерживаемыми режимами анализа;
- запуск анализа проходит через `IAiAnalysisEngine`;
- результат external AI/RAG режима должен сохранять retrieved context или metadata, если они доступны;
- отсутствие retrieved context должно явно фиксироваться и отображаться пользователю;
- сведения о режиме анализа, analysis engine и provider/adapter должны сохраняться и попадать в экспорт;
- Markdown и JSON export должны быть расширены сведениями о режиме анализа и основании результата;
- экспертная оценка и экспертное заключение остаются отдельным человеческим слоем;
- результаты direct LLM и external AI/RAG режимов должны быть сопоставимы для экспертной оценки и будущей экспериментальной проверки;
- MVP-1 не включает собственный RAG, интеграции с рабочими системами, workflow, dashboard или PDF.
