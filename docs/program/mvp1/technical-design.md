# Технический проект SDD MVP-1 программы анализа влияния проектных запросов

## Назначение документа

Документ фиксирует содержательный технический проект SDD MVP-1 программы анализа влияния проектных запросов.

Документ подготовлен на основе:

- `docs/program/mvp1/mvp1-strategy.md`;
- `docs/program/mvp1/clarification-decisions.md`;
- `docs/program/mvp1/requirements-draft.md`;
- технических рамок MVP-0, зафиксированных в `docs/program/mvp0/technical-design.md`.

Документ не является реализацией, implementation plan, task breakdown, тестовым планом или описанием конкретных задач разработки. Его задача - описать архитектуру MVP-1, основные технические границы, модель подключения external AI/RAG режима, сохранение результата и расширение экспорта без превращения проекта в собственную RAG-платформу.

MVP-1 рассматривается как развитие MVP-0. Он сохраняет существующий контур direct LLM analysis и добавляет второй режим интеллектуального анализа через внешний AI/RAG-контур. Архитектурный центр остается внутри собственной программы: карточка анализа, сохранение исходных данных, представление результата, экспертная оценка, экспертное заключение и экспорт принадлежат приложению. Внешний AI/RAG-движок выступает сменяемым provider/adapter.

## Цели технического проекта MVP-1

Технический проект MVP-1 должен обеспечить:

- сохранение возможностей MVP-0 по созданию анализа, manual context, запуску анализа, сохранению результата, экспертной оценке, экспертному заключению и экспорту;
- поддержку двух режимов анализа: direct LLM и external AI/RAG;
- подключение `ExternalRagAnalysisEngine` за существующей application-level границей `IAiAnalysisEngine`;
- сохранение `DirectLlmAnalysisEngine` как отдельного поддерживаемого режима;
- выбор режима анализа до запуска и фиксацию выбранного режима в сохраненном результате;
- нейтральный контракт external AI/RAG adapter, не привязанный к Dify или другому конкретному продукту;
- минимальную модель retrieved context и retrieved context metadata;
- возможность использовать Dify как первую практическую реализацию нейтрального adapter;
- приведение результата external RAG к существующей общей логике `ImpactMap`;
- сохранение metadata, engine mode, retrieved context и ограниченных диагностических сведений по ответу; полный raw response external AI/RAG provider не сохраняется по умолчанию;
- расширение Markdown и JSON export сведениями о режиме анализа, provider/adapter и основании результата;
- сохранение запрета на прямые вызовы Dify, retrieval API, RAG или внешних LLM из UI/Razor Pages;
- явное отделение AI/RAG/LLM результата от экспертного решения человека.

Технический проект не должен проектировать собственную RAG-платформу, собственные embeddings, vector database, rerank, agentic search или перенос retrieval pipeline внутрь приложения.

## Архитектурное решение MVP-1

MVP-1 сохраняет архитектурный стиль MVP-0: локальное web-приложение в стиле модульного монолита.

Архитектура развивается за счет добавления второго analysis engine и нижнего external AI/RAG adapter boundary.

```text
---------------------------------------------------------------+
| Web UI / Razor Pages                                         |
| - анализы, context, выбор режима, просмотр результата, export|
+---------------------------------------------------------------+
| Application Services                                         |
| - сценарии анализа, выбор engine, сохранение, оценка, export |
+---------------------------------------------------------------+
| Application AI Boundary                                      |
| - IAiAnalysisEngine                                          |
| - DirectLlmAnalysisEngine                                    |
| - ExternalRagAnalysisEngine                                  |
+---------------------------------------------------------------+
| Domain Model                                                 |
| - Analysis, AnalysisRun, ImpactMap, ExpertEvaluation         |
| - ManualContext, RetrievedContext                            |
+---------------------------------------------------------------+
| Infrastructure                                               |
| - SQLite, file storage, LLM provider, external AI/RAG adapter|
| - Markdown/JSON exporters                                    |
+---------------------------------------------------------------+
| External AI/RAG Engine                                       |
| - Dify or another external provider                          |
| - retrieval and retrieval-assisted analysis                  |
+---------------------------------------------------------------+
```

Главная граница интеллектуального анализа остается `IAiAnalysisEngine`. UI, Razor Pages, page models, controllers и доменная логика не вызывают Dify, retrieval API, внешний RAG или внешний LLM напрямую. Они передают пользовательский выбор режима и входные данные в application service, который выбирает соответствующую реализацию `IAiAnalysisEngine`.

## Сохранение MVP-0 и DirectLlmAnalysisEngine

`DirectLlmAnalysisEngine` сохраняется как полноценный режим анализа. Он не удаляется, не считается устаревшим и не подменяется external RAG режимом.

В MVP-1 direct LLM режим продолжает выполнять следующие функции:

- принимать исходные данные анализа;
- использовать manual context, добавленный пользователем внутри программы;
- формировать запрос к LLM через существующий provider boundary;
- возвращать структурированный результат в общей логике `ImpactMap`;
- сохранять metadata выполнения и доступные диагностические сведения в согласованном для режима объеме;
- участвовать в Markdown и JSON export.

Для ранее созданных анализов MVP-0, где сведения о режиме анализа могли отсутствовать, приложение должно сохранять обратную совместимость. Такие результаты могут отображаться как ранее созданные direct LLM результаты без полной metadata MVP-1, а не как ошибочные или поврежденные данные.

## ExternalRagAnalysisEngine за IAiAnalysisEngine

`ExternalRagAnalysisEngine` добавляется как новая реализация `IAiAnalysisEngine`.

Его ответственность:

- принять стандартный вход интеллектуального анализа от application service;
- сформировать нейтральный запрос к external AI/RAG adapter;
- передать во внешний контур исходные данные анализа и, если выбран соответствующий вариант запуска, manual context как отдельный входной блок;
- получить retrieval-assisted аналитический результат;
- получить retrieved context или metadata, если они доступны;
- сохранить provider metadata и sanitized diagnostic snapshot или metadata response внешнего adapter по умолчанию;
- привести аналитический результат к общей структуре `ImpactMap`;
- вернуть application service единый результат анализа с mode, engine metadata, provider metadata и retrieved context state.

`ExternalRagAnalysisEngine` не реализует retrieval pipeline внутри приложения. Он не создает embeddings, не обращается к собственной vector database, не выполняет rerank и не координирует многошаговый agentic search внутри приложения. Эти действия, если они есть, остаются внутри внешнего AI/RAG-движка.

## Выбор режима анализа

В MVP-1 режим анализа выбирается до запуска интеллектуального анализа.

Минимальные режимы:

- `DirectLlm`;
- `ExternalRag`.

Выбор режима фиксируется в application-level request на запуск анализа и сохраняется вместе с результатом.

Технически выбор режима должен проходить через application service:

1. UI показывает пользователю доступные режимы.
2. Пользователь выбирает режим.
3. Page model передает выбранный режим в application service как часть команды запуска анализа.
4. Application service проверяет входные данные и доступность выбранного режима.
5. Application service выбирает соответствующий `IAiAnalysisEngine`.
6. Выбранный engine выполняет анализ.
7. Application service сохраняет результат, mode, metadata, диагностические сведения согласованного объема и основание результата.

UI не должен знать, какой SDK, endpoint или provider используется внутри engine. UI работает только с режимом анализа и результатом, сохраненным приложением.

Для выбора engine целесообразно использовать application-level registry или selector, который сопоставляет `AnalysisMode` с реализацией `IAiAnalysisEngine`. Это сохраняет возможность добавить новый external provider без изменения Razor Pages и доменной логики.

Недоступность external AI/RAG режима не должна ломать direct LLM сценарий. Пользователь должен иметь возможность продолжить работу в direct LLM режиме, если он доступен и применим.

## Нейтральный контракт external AI/RAG adapter

Для подключения внешних AI/RAG-движков вводится нейтральный adapter boundary. Этот boundary находится ниже `ExternalRagAnalysisEngine` и не является публичным UI/API приложения.

Назначение adapter boundary - скрыть особенности конкретного внешнего продукта и вернуть программе результат в согласованной форме.

### Вход external AI/RAG adapter

Нейтральный запрос должен включать:

- идентификатор анализа или correlation id для диагностики;
- исходные данные анализа:
  - название или краткое описание запроса;
  - исходное требование или описание текущего состояния;
  - предлагаемое изменение;
  - описание ситуации;
  - источник запроса или изменения;
- manual context как отдельный optional block, если пользователь разрешил передать его во внешний контур;
- указание ожидаемой структуры результата, совместимой с `ImpactMap`;
- указание, что результат является предварительным аналитическим материалом, а экспертное решение принимает человек;
- технические execution metadata, не содержащие секретов:
  - requested mode;
  - adapter name;
  - correlation id;
  - timestamp;
  - optional prompt/profile version.

Контракт не должен требовать, чтобы внешний движок раскрывал внутренний retrieval pipeline. Для приложения достаточно получить аналитический результат и доступное основание результата.

### Выход external AI/RAG adapter

Нейтральный ответ должен включать:

- статус выполнения:
  - completed;
  - completedWithWarnings;
  - failed;
  - partial;
- структурированный аналитический результат или данные, пригодные для приведения к `ImpactMap`;
- по умолчанию только sanitized diagnostic snapshot, metadata response или ограниченные диагностические сведения; полный raw response не сохраняется и допускается только по отдельному явному решению или в debug/manual режиме;
- provider metadata:
  - adapter name;
  - provider name;
  - external workflow/application id, если применимо;
  - model name, если доступно и не является секретом;
  - execution timestamp;
  - request/correlation id, если доступен;
- retrieved context state:
  - available;
  - metadataOnly;
  - unavailable;
  - partial;
- список retrieved context items, если они доступны;
- warnings об ограничениях результата;
- error information, если анализ не был успешно завершен.

Контракт не должен быть Dify-specific. Названия полей в реализации могут уточняться, но смысловая модель должна оставаться применимой к другим внешним AI/RAG-движкам.

## Минимальная модель retrieved context

Retrieved context хранится отдельно от manual context.

Manual context - это фрагменты, которые пользователь добавил в программу вручную или через подготовленные файлы.

Retrieved context - это фрагменты или metadata, найденные или выбранные внешним AI/RAG-движком во время external AI/RAG анализа.

Минимальная логическая модель retrieved context item:

- id внутри результата анализа;
- analysis run id или ссылка на результат анализа;
- source title;
- source id или external source reference;
- fragment id или chunk id, если доступен;
- text или excerpt, если внешний движок возвращает текст;
- url или external reference, если доступен;
- rank, если доступен;
- score или аналог релевантности, если доступен;
- provider/adapter, который вернул фрагмент;
- metadata в ограниченном key-value виде для дополнительных сведений;
- признак полноты:
  - fullText;
  - excerptOnly;
  - metadataOnly;
  - unavailable;
- warning или limitation note, если фрагмент неполон.

Для MVP-1 не требуется хранить весь внутренний retrieval trace внешнего движка. Достаточно сохранить то основание результата, которое внешний движок явно вернул и которое может быть показано человеку или экспортировано.

Retrieved context не редактируется экспертом как manual context. Он используется как сохраненное основание внешнего AI/RAG результата и как материал для проверки предварительного аналитического вывода.

## Retrieved context metadata и ограничения полноты

External AI/RAG adapter может вернуть разные уровни основания результата.

### Полный retrieved context

Если внешний движок вернул текст или excerpt фрагментов, приложение сохраняет retrieved context items и показывает их отдельно от manual context.

### Только metadata

Если внешний движок вернул source title, source id, fragment id, rank, score или ссылки без текста фрагментов, приложение сохраняет эти metadata и явно показывает, что полный текст retrieved context недоступен.

### Отсутствие retrieved context

Если внешний движок вернул аналитический результат, но не вернул retrieved context и не вернул пригодные metadata, приложение сохраняет результат, но фиксирует ограничение воспроизводимости: основание внешнего retrieval недоступно для просмотра внутри программы.

### Частичный retrieved context

Если часть фрагментов доступна, а часть отсутствует или представлена только metadata, приложение сохраняет доступные элементы и фиксирует общий state результата как partial.

Ограничение retrieved context не должно смешиваться с экспертным заключением. Это техническое и аналитическое ограничение основания AI/RAG результата, а не вывод эксперта.

## Dify adapter как первая реализация

Dify может быть первой практической реализацией external AI/RAG adapter для MVP-1.

Роль Dify adapter:

- принять нейтральный request от `ExternalRagAnalysisEngine`;
- преобразовать его в формат вызова Dify workflow/application;
- вызвать Dify через инфраструктурный provider boundary;
- получить ответ Dify;
- извлечь из ответа аналитический результат, provider metadata, retrieved context metadata и доступные диагностические сведения согласованного объема, если они доступны;
- преобразовать ответ Dify обратно в нейтральный external AI/RAG response.

Dify adapter не должен определять публичную архитектуру приложения. Его особенности не должны протекать в UI, Razor Pages, доменную модель или export как обязательная структура.

Dify-specific структура запроса и ответа не становится стабильной публичной моделью приложения. В сохраненном результате и export могут фиксироваться provider/adapter name, общая metadata, warnings и sanitized diagnostic snapshot или metadata response без секретов, токенов, персональных данных и чувствительных материалов, но смысловая модель программы должна оставаться нейтральной и применимой к другому external AI/RAG provider.

Если Dify не возвращает полный retrieved context, adapter должен вернуть `metadataOnly`, `partial` или `unavailable` state, а не скрывать ограничение.

Если в дальнейшем будет выбран другой внешний AI/RAG-движок, он должен подключаться через тот же нейтральный adapter boundary без изменения сценариев UI и без нарушения `IAiAnalysisEngine`.

## Приведение external RAG результата к ImpactMap

Результат external AI/RAG режима должен быть приведен к общей логике `ImpactMap`, используемой для direct LLM режима и экспертной оценки.

Смысловые разделы `ImpactMap` сохраняются общими:

- краткое описание изменения;
- потенциально затронутые требования;
- потенциально затронутые задачи;
- потенциально затронутые проектные решения;
- потенциально затронутые API, интерфейсы, документы и тесты;
- потенциально затронутые архитектурные ограничения;
- потенциально затронутые элементы организационного контекста;
- противоречия;
- недостающая информация;
- вопросы для уточнения;
- риски;
- варианты для экспертного рассмотрения;
- предварительная аналитическая оценка.

Для external AI/RAG результата допускается дополнительная связь элементов `ImpactMap` с retrieved context item ids, если внешний движок вернул такие связи или если adapter может надежно их сопоставить.

Эта связь является полезной, но не обязательной для каждого элемента. Если связь отсутствует, элемент карты влияния сохраняется без искусственной ссылки на retrieved context. Программа не должна создавать ложную трассируемость.

Если внешний response содержит структуру, отличающуюся от `ImpactMap`, `ExternalRagAnalysisEngine` выполняет normalization: переносит доступные выводы в общие разделы, сохраняет sanitized diagnostic snapshot или metadata response по умолчанию и фиксирует warning о частичной нормализации, если часть данных не удалось сопоставить.

## Данные результата анализа

Для MVP-1 целесообразно рассматривать результат интеллектуального анализа как отдельный analysis run или аналогичный логический объект. Конкретная структура таблиц определяется позже, но технический проект фиксирует минимальный состав сохраняемых данных.

Термин `analysis run` в этом документе используется как логическое обозначение сохраненного результата запуска анализа. Он не означает обязательного введения отдельной таблицы, полной истории запусков или переписывания модели хранения MVP-0. На этапе implementation planning должна быть выбрана минимальная совместимая модель расширения существующего хранения, при которой ранее созданные результаты MVP-0 остаются пригодными для просмотра и экспорта.

Сохраняемый результат анализа должен включать:

- id результата или запуска;
- analysis id;
- status;
- generatedAt;
- analysisMode;
- engineName;
- providerName, если применимо;
- adapterName, если применимо;
- modelName или workflowName, если доступно и допустимо;
- promptVersion или requestProfileVersion, если применимо;
- inputSnapshot;
- diagnosticSnapshot или metadataResponse;
- normalizedResult;
- impactMap;
- retrievedContextState;
- retrievedContextItems или retrievedContextMetadata;
- warnings;
- errorMessage, если анализ завершился ошибкой;
- сведения о том, использовался ли manual context во внешнем AI/RAG запросе.

`analysisMode` является обязательной metadata MVP-1. Для старых результатов MVP-0, где mode не сохранен, приложение должно отображать legacy state как отсутствие полной metadata MVP-1 у ранее созданного результата.

## Хранение diagnostic snapshot, metadata, mode и retrieved context

Для external AI/RAG режима по умолчанию сохраняются не полный raw response, а ограниченные диагностические сведения:

- sanitized diagnostic snapshot;
- metadata response;
- иные ограниченные диагностические сведения, достаточные для воспроизводимости и диагностики.

Эти диагностические сведения не должны содержать секреты, токены, персональные данные и чувствительные материалы. Полный raw response external AI/RAG provider может сохраняться только по отдельному явному решению или в debug/manual режиме.

Metadata сохраняется отдельно от raw response или diagnostic snapshot, чтобы приложение могло показывать ключевые сведения без разбора provider-specific payload.

Минимальная metadata результата:

- analysis mode;
- engine name;
- provider name;
- adapter name, если применимо;
- model/workflow/profile name, если доступно;
- generatedAt;
- status;
- retrieved context state;
- warnings;
- indication whether manual context was sent to external provider.

Retrieved context сохраняется отдельно от manual context и отдельно от normalized `ImpactMap`. Это позволяет:

- показывать retrieved context как основание external AI/RAG результата;
- экспортировать retrieved context независимо от ручного контекста;
- фиксировать ограничение, если external provider не вернул основание;
- сравнивать direct LLM и external AI/RAG режимы без смешения источников контекста.

## Изменения в Markdown export

Markdown export MVP-1 должен оставаться человекочитаемым и расширяться сведениями о режиме анализа и основании результата.

В Markdown export добавляются разделы или блоки:

- режим анализа;
- analysis engine;
- provider/adapter, если применялся;
- сведения о том, был ли manual context передан во внешний AI/RAG контур;
- retrieved context для external AI/RAG режима, если доступен;
- retrieved context metadata, если полный текст недоступен;
- явное ограничение, если retrieved context отсутствует или неполон;
- предупреждения analysis engine или adapter, если они есть.

Для direct LLM режима Markdown export не должен создавать искусственный раздел retrieved context. Он может явно указать, что результат получен в direct LLM режиме и использовал manual context, если он был добавлен.

Для external AI/RAG режима Markdown export должен показывать основание результата в отдельном разделе, например:

- "Retrieved context";
- "Retrieved context metadata";
- "Ограничения основания external AI/RAG результата".

Формулировки должны сохранять позицию проекта: AI/RAG/LLM формирует предварительный аналитический материал, а экспертное заключение остается за человеком.

## Изменения в JSON export

JSON export MVP-1 должен быть пригоден для последующего сравнения direct LLM и external AI/RAG режимов.

В JSON export добавляются или расширяются следующие смысловые блоки:

- `analysisMode`;
- `analysisEngine`;
- `provider`;
- `adapter`;
- `inputSnapshot`;
- `manualContext`;
- `manualContextUsage`;
- `aiAnalysisResult`;
- `impactMap`;
- `retrievedContext`;
- `retrievedContextState`;
- `retrievedContextLimitations`;
- `rawResponseMetadata`;
- `warnings`;
- `expertEvaluation`;
- `expertConclusion`;
- `exportMetadata`.

Конкретная JSON Schema не фиксируется этим документом как отдельный стандарт, но имена смысловых блоков должны быть стабильными после начала реализации, чтобы результаты можно было использовать в исследовательском сравнении.

JSON export должен позволять отличить:

- результат direct LLM без retrieved context;
- external AI/RAG результат с полным retrieved context;
- external AI/RAG результат только с metadata;
- external AI/RAG результат без доступного retrieved context;
- legacy результат MVP-0 без полной metadata MVP-1.

В стабильный JSON export не должен включаться полный Dify-specific или иной provider-specific raw response как обязательная структура. Для воспроизводимости в export достаточно metadata raw response, provider/adapter сведений и ограниченных диагностических сведений согласованного объема. Полный raw response может включаться только по отдельному явному решению или в debug/manual режиме.

JSON export не должен требовать от потребителя знания Dify-specific схемы. Стабильными считаются только общие смысловые блоки результата, режима анализа, metadata, retrieved context state, expert evaluation и expert conclusion.

## UI и запрет прямых вызовов Dify/RAG

UI MVP-1 остается набором Razor Pages, как в MVP-0.

UI может:

- показывать выбор режима анализа;
- показывать доступность external AI/RAG режима;
- показывать сведения о выбранном режиме;
- показывать retrieved context или ограничение его отсутствия;
- показывать warnings external adapter;
- запускать application-level команду анализа.

UI не должен:

- вызывать Dify API напрямую;
- хранить Dify-specific request/response logic;
- знать endpoint внешнего RAG-движка;
- формировать provider-specific payload;
- обращаться к retrieval API;
- выполнять embeddings, rerank или поиск по внешней базе знаний.

Page model должен передавать в application service только пользовательские входные данные, выбранный режим и параметры запуска, допустимые на application level. Все provider-specific детали остаются в infrastructure adapter.

Для предотвращения утечки внешней интеграции в UI вводятся архитектурные правила:

- Razor Pages зависят от application services, а не от external AI/RAG adapters;
- external adapter interfaces размещаются в инфраструктурной или application-adapter области, недоступной для UI как сценарный dependency;
- `ExternalRagAnalysisEngine` является единственной точкой, которая обращается к external AI/RAG adapter;
- export строится из сохраненной модели результата, а не из повторного вызова provider.

## Ошибки и неполные ответы external AI/RAG

External AI/RAG режим должен корректно обрабатывать неполные и ошибочные ответы.

Возможные состояния:

- внешний adapter недоступен;
- внешний adapter вернул ошибку;
- внешний adapter вернул диагностический ответ или metadata, но не вернул структурированный результат;
- внешний adapter вернул структурированный результат без retrieved context;
- внешний adapter вернул retrieved context metadata без текста;
- внешний adapter вернул результат, который удалось только частично привести к `ImpactMap`.

В случае ошибки приложение сохраняет status failed и диагностическое сообщение без раскрытия секретов.

В случае частичного результата приложение может сохранить просмотр результата, если он пригоден для экспертной оценки, но должно зафиксировать warning и limitation.

Неполнота retrieved context не должна блокировать сохранение аналитического результата, если сам результат пригоден для просмотра. Но такая неполнота должна быть видна в UI и export.

## Безопасность и конфиденциальность

MVP-1 сохраняет демонстрационный и исследовательский характер проекта.

Минимальные решения:

- секреты внешнего AI/RAG provider не хранятся в репозитории;
- настройки provider задаются через конфигурацию окружения или другой согласованный механизм секретов;
- UI должен дать пользователю понять, что external AI/RAG режим передает данные во внешний контур;
- manual context передается во внешний AI/RAG контур только при явном выборе такого поведения для запуска;
- для проверки Dify adapter используются подготовленные или обезличенные материалы;
- по умолчанию сохраняются только sanitized diagnostic snapshot, metadata response или иные ограниченные диагностические сведения без секретов, токенов, персональных данных и чувствительных материалов; полный raw response допустим только по отдельному явному решению или в debug/manual режиме.

MVP-1 не подключает реальные корпоративные источники напрямую. Если внешний AI/RAG-движок подключен к корпоративной базе знаний, ответственность за retrieval и доступ к этой базе находится во внешнем контуре, а приложение сохраняет только возвращенный результат и доступные metadata.

## Сопоставимость direct LLM и external AI/RAG

Сопоставимость режимов обеспечивается общей структурой результата и export.

Оба режима должны сохранять:

- исходные данные анализа;
- сведения о mode и engine;
- structured analysis result;
- `ImpactMap`;
- expert evaluation;
- expert conclusion;
- status and timestamps;
- по умолчанию только sanitized diagnostic snapshot, metadata response или ограниченные диагностические сведения; полный raw response не является штатной моделью хранения и допускается только по отдельному явному решению или в debug/manual режиме;
- export metadata.

Отличия режимов сохраняются явно:

- direct LLM использует manual context и не имеет retrieved context;
- external AI/RAG может иметь retrieved context, metadata или limitation;
- external AI/RAG может использовать manual context только как отдельный optional input, если пользователь это выбрал;
- provider/adapter metadata для external AI/RAG должна быть видна в сохраненном результате и export.

Сопоставимость не означает одинаковость результатов. Она означает, что эксперт и исследователь могут сравнить результаты по общим смысловым разделам карты влияния и экспертной оценки.

## Экспертное решение

Экспертная оценка и экспертное заключение остаются отдельным человеческим слоем.

AI/RAG/LLM результат не утверждает изменение, не отклоняет его, не назначает исполнителей, не создает задачи и не изменяет требования, документы, API или код.

Эксперт может:

- подтвердить выводы AI/RAG/LLM;
- скорректировать выводы;
- отклонить выводы;
- указать пропущенные элементы;
- зафиксировать ограничение применимости результата;
- сформировать итоговое экспертное заключение.

Retrieved context является основанием предварительного external AI/RAG результата, но не заменяет экспертное основание итогового заключения.

## Что откладывается

В MVP-1 не включается:

- собственная RAG-платформа;
- собственные embeddings;
- собственная vector database;
- собственный rerank;
- собственный retrieval pipeline;
- agentic search;
- agentic workflow;
- интеграции с Jira, Confluence, GitLab, ALM, почтой, чатами или workflow-системами;
- dashboard для управления задачами или исследовательских метрик;
- автоматическое создание задач, pull request или изменений в документах;
- автоматическое утверждение, отклонение или эскалация проектного запроса;
- превращение Dify в обязательную основу архитектуры;
- детальный task breakdown реализации.

## Риски технического проекта MVP-1

Основные риски:

- внешний AI/RAG-движок может вернуть неполный retrieved context или не вернуть его совсем;
- внешний AI/RAG-движок может вернуть результат, плохо приводимый к `ImpactMap`;
- Dify-specific детали могут протечь в доменную модель, UI или export;
- пользователь может воспринять retrieved context как исчерпывающее основание управленческого решения;
- external AI/RAG режим может создать ложное ощущение большей точности результата;
- raw response может содержать чувствительные данные или provider-specific служебную информацию;
- сохранение нескольких режимов анализа может усложнить модель результата;
- JSON export может стать нестабильным, если заранее не выделить общие смысловые блоки;
- передача manual context во внешний контур может быть нежелательной для конфиденциальных материалов;
- обработка partial response может усложнить UI и экспертную интерпретацию.

Снижение рисков:

- сохранять retrieved context state и явно показывать ограничения;
- по умолчанию сохранять sanitized diagnostic snapshot или metadata response отдельно от normalized result и не сохранять полный raw response external AI/RAG provider;
- держать Dify за нейтральным adapter boundary;
- не выводить Dify-specific поля в UI как обязательную модель программы;
- приводить external AI/RAG результат к общей `ImpactMap`, сохраняя warnings при частичной нормализации;
- не создавать искусственные ссылки элементов карты влияния на retrieved context;
- явно отделять AI/RAG результат от экспертного заключения;
- использовать подготовленные или обезличенные данные для демонстрации;
- стабилизировать смысловые блоки JSON export до реализации;
- передавать manual context во внешний контур только по явному выбору пользователя.
