# MVP-3 UI flow анализа влияния проектного изменения

## Назначение документа

Документ описывает целевой пользовательский сценарий MVP-3 и текстовую структуру экранов для программы анализа влияния проектных изменений.

Документ не является техническим проектом, implementation plan, описанием визуального дизайна, набором UI-компонентов или разрешением на реализацию. Его задача - зафиксировать логику пользовательского пути так, чтобы она не противоречила уже реализованному контуру MVP-0/MVP-2 и сохраняла исследовательскую позицию проекта: AI/RAG/LLM формирует предварительный аналитический материал, а экспертное заключение фиксирует человек.

Основанием для документа являются:

- `docs/workflow/README.md`;
- `docs/workflow/checklists.md`;
- `docs/workflow/development-cycle.md`;
- `docs/program/mvp0/initial-concept.md`;
- `docs/program/mvp0/clarification-decisions.md`;
- `docs/program/mvp0/requirements-draft.md`;
- `docs/program/mvp0/technical-design.md`;
- `docs/program/mvp0/implementation-plan.md`;
- `docs/program/mvp0/ui-concept.md`;
- документы `docs/program/mvp1/` и `docs/program/mvp2/`;
- текущая структура Razor Pages UI на уровне чтения.

Термины контекста в этом документе разделяются строго:

- manual context - материалы, которые пользователь явно добавляет в карточку анализа;
- RAG knowledge base - внешний корпус или база знаний external AI/RAG контура/provider-а, а не новая база знаний приложения;
- retrieved context - сохраненный результат извлечения для конкретного анализа: фактически возвращенные provider-ом фрагменты и/или metadata, которые могут быть полными, partial, metadata-only или unavailable.

## Контекст MVP-3

MVP-3 рассматривается как развитие уже существующего пользовательского контура анализа, а не как новая система управления задачами, база знаний, workflow согласования или чат-бот.

В текущем приложении уже есть основные смысловые зоны:

- список сохраненных анализов;
- создание анализа;
- карточка деталей анализа;
- ручной контекст;
- проверка входных данных перед запуском анализа;
- выбор режима `Direct LLM` или `External AI/RAG`;
- предварительный результат AI-анализа;
- карта влияния;
- основания external AI/RAG результата и ограничения retrieved context;
- экспертная оценка;
- экспертное заключение;
- экспорт Markdown/JSON.

MVP-3 должен описывать эти зоны как цельный пользовательский flow и уточнять, как пользователь проходит путь от проектного изменения к экспертному заключению и экспорту. При этом документ не фиксирует, должны ли зоны быть отдельными страницами, шагами мастера или блоками одного рабочего пространства.

## Принципы UI flow

1. Пользователь работает с одним артефактом: анализом влияния одного проектного изменения.
2. Исходное изменение, проектный контекст, предварительный AI/RAG/LLM-результат, экспертная оценка и экспертное заключение отображаются как разные смысловые слои.
3. Выбор `Direct LLM` или `External AI/RAG` является выбором режима подготовки аналитического материала, а не выбором субъекта управленческого решения.
4. Для external AI/RAG результата пользователь должен видеть доступные основания результата и ограничения воспроизводимости, включая частичный, metadata-only или отсутствующий retrieved context.
5. Карта влияния остается основной формой результата анализа; свободный ответ модели может использоваться только как вспомогательный диагностический материал.
6. Экспертная оценка и экспертное заключение выполняются человеком после просмотра карты влияния, оснований и ограничений.
7. Экспорт Markdown/JSON выполняется из сохраненного артефакта анализа и должен сохранять разделение предварительного AI-результата и экспертного заключения.

## Целевой пользовательский сценарий

1. Пользователь открывает список анализов.
2. Пользователь создает новый анализ влияния проектного изменения.
3. Пользователь выбирает тип проектного запроса.
4. Пользователь заполняет текущее состояние: исходное описание, требование, ограничение, API, интеграцию, проектное решение или другой релевантный фрагмент. Это входной снимок, заданный пользователем; он не считается RAG knowledge base и не подменяет retrieved context.
5. Пользователь заполняет проектное изменение: предлагаемое изменение требования, новой функциональности, bug fix, уточнение, изменение API/интеграции, технического ограничения или проектного решения.
6. Пользователь указывает ситуацию и источник изменения.
7. При необходимости пользователь добавляет ручной контекст: задачи, фрагменты документов, комментарии, API-описания, архитектурные ограничения, тестовые сценарии, ранее принятые решения или другие материалы.
8. Пользователь запускает анализ влияния.
9. Перед запуском или в момент запуска пользователь выбирает режим подготовки предварительного результата: `Direct LLM` или `External AI/RAG`.
10. Программа формирует предварительный аналитический материал и показывает карту влияния.
11. Пользователь проверяет основания и ограничения результата: входной снимок данных, ручной контекст, режим анализа, provider/adapter metadata, предупреждения, retrieved context и ограничения external AI/RAG, если они доступны.
12. Эксперт выполняет экспертную оценку: подтверждает, корректирует, отклоняет или помечает как требующие уточнения элементы карты влияния, фиксирует пропущенные элементы, дополнительные исправления, достаточность контекста и полезность результата.
13. Эксперт фиксирует экспертное заключение: тип заключения, комментарий и обоснование.
14. Пользователь экспортирует результат в Markdown или JSON для обсуждения, экспериментальной проверки или включения в исследовательские материалы.

## Смысловые состояния артефакта

Состояния отражают готовность артефакта анализа, а не маршрут согласования:

- анализ создан, входные данные еще заполняются;
- минимальные входные данные заполнены;
- контекст добавлен или осознанно отсутствует;
- входные данные проверены перед запуском;
- предварительный анализ запущен;
- предварительный результат получен или получен с предупреждениями;
- предварительный результат не получен или требует диагностики;
- карта влияния доступна для экспертной оценки;
- экспертная оценка сохранена;
- экспертное заключение зафиксировано;
- результат экспортирован.

Эти состояния не должны запускать автоматическое создание задач, уведомления, маршруты утверждения, изменение документации, изменение кода или действия во внешних системах.

## Текстовая структура экранов

### Список анализов

Назначение: входная точка для работы с сохраненными анализами и создания нового анализа.

Содержит:

- название списка сохраненных анализов;
- пустое состояние, если анализов еще нет;
- перечень анализов;
- название анализа;
- статус артефакта;
- дату последнего обновления;
- переход к проверке входных данных;
- переход к деталям анализа;
- действие создания нового анализа.

Логика экрана:

- пользователь должен быстро понять, какие анализы уже существуют и на каком этапе находится каждый артефакт;
- список не является task board, backlog или workflow-панелью;
- статус показывает состояние анализа, а не управленческий маршрут.

### Создание анализа

Назначение: зафиксировать основу анализа влияния проектного изменения.

Содержит:

- название анализа;
- тип проектного запроса;
- текущее состояние;
- проектное изменение;
- описание ситуации;
- источник изменения;
- сохранение анализа;
- возврат к списку анализов.

Логика экрана:

- тип проектного запроса помогает пользователю описать природу изменения, но не должен превращаться в жесткую отраслевую классификацию;
- текущее состояние и проектное изменение должны быть разделены, чтобы анализ не смешивал "как сейчас" и "что предлагается изменить";
- текущее состояние является пользовательским входным снимком для конкретного анализа, а не внешней RAG knowledge base и не retrieved context;
- ситуация и источник изменения нужны для дальнейшей экспертной проверки и экспортируемого отчета;
- на этом этапе не показываются выводы AI/RAG/LLM и не фиксируется экспертное заключение.

Рекомендуемые типы проектного запроса на уровне смысла:

- изменение требования;
- новая функциональность;
- исправление ошибки;
- уточнение;
- изменение API или интеграции;
- изменение технического ограничения;
- изменение проектного решения;
- другое проектное изменение.

### Детали анализа

Назначение: показать сохраненный артефакт анализа как рабочее пространство для перехода к контексту, запуску анализа, просмотру результата, экспертной оценке, заключению и экспорту.

Содержит:

- название анализа;
- идентификатор;
- даты создания, обновления и фиксации;
- текущий статус;
- тип проектного запроса;
- текущее состояние;
- проектное изменение;
- описание ситуации;
- источник изменения;
- ручной контекст;
- предварительный результат AI-анализа, если он уже получен;
- карту влияния, если она сохранена;
- экспертное заключение, если оно зафиксировано;
- действия перехода к проверке ввода, экспертной оценке, экспертному заключению и экспорту.

Логика экрана:

- детали анализа являются обзором сохраненного артефакта, а не страницей принятия автоматического решения;
- пользователь должен видеть, какие данные были основанием для анализа;
- после экспертной фиксации пользователь должен ясно видеть, что заключение принадлежит человеку;
- экспорт доступен только как выгрузка сохраненного результата, а не как новый анализ.

### Предварительный результат

Назначение: показать результат, полученный через выбранный analysis engine, как предварительный аналитический материал.

Содержит:

- статус результата;
- дату формирования;
- режим анализа: `Direct LLM` или `External AI/RAG`;
- имя механизма, provider, adapter или профиля, если эти сведения доступны;
- версию prompt или профиля, если она сохраняется;
- сведения о ручном контексте и его передаче во внешний контур;
- предупреждения и ограничения;
- sanitized raw analysis text или ограниченный диагностический текст, если он уже сохранен для воспроизводимости и не содержит секретов;
- снимок входных данных.

Логика экрана:

- результат должен быть явно маркирован как предварительный аналитический материал;
- предупреждения модели, adapter-а или validator-а не скрываются;
- sanitized raw analysis text не заменяет структурированную карту влияния;
- полный provider-specific raw payload не является публичной UI-моделью;
- диагностика не должна раскрывать API keys, bearer tokens, cookies, CSRF, passwords или чувствительный provider payload.

### Карта влияния

Назначение: представить структурированную карту влияния проектного изменения для последующей экспертной оценки.

Содержит:

- краткую сводку изменения;
- потенциально затронутые требования;
- потенциально затронутые задачи;
- потенциально затронутые проектные решения;
- потенциально затронутые API, интерфейсы, документы и тесты;
- потенциально затронутые архитектурные ограничения;
- потенциально затронутые элементы организационного контекста;
- противоречия;
- недостающую информацию;
- вопросы для уточнения;
- риски;
- варианты для экспертного рассмотрения;
- предварительную аналитическую оценку;
- стабильные идентификаторы элементов карты, если они используются для экспертной оценки и JSON-экспорта.

Логика экрана:

- карта влияния является главным представлением AI/RAG/LLM-результата;
- элементы карты не являются управленческими решениями;
- критичность и тип элемента помогают эксперту проверить результат, но не должны автоматически запускать действия;
- пустые секции допустимы и должны отображаться честно.

### Основания и ограничения

Назначение: показать, на каких данных основан предварительный результат и какие ограничения нужно учитывать при экспертной проверке.

Содержит:

- исходные данные анализа;
- ручной контекст;
- сведения о выбранном режиме анализа;
- сведения о передаче ручного контекста во внешний AI/RAG-контур, если применимо;
- retrieved context state для external AI/RAG результата;
- список retrieved context items, если они сохранены;
- source title, source id, fragment id, external reference, URL/reference, rank, score и metadata, если эти сведения доступны;
- сохраненный текст или выдержку источника, если они доступны;
- предупреждение, если сохранены только metadata, частичная выдержка или retrieved context отсутствует;
- limitation note по воспроизводимости результата.

Логика экрана:

- `Direct LLM` результат опирается на входной снимок данных и ручной контекст, переданный в анализ;
- `External AI/RAG` результат дополнительно требует честного отображения retrieved context и его полноты;
- RAG knowledge base остается внешним корпусом provider-а; UI показывает только retrieved context, фактически возвращенный и сохраненный для конкретного анализа;
- отсутствие retrieved context не должно маскироваться и не должно подменяться выдуманными источниками;
- external AI/RAG provider или Dify-specific детали не должны становиться публичной UI-моделью сверх безопасных нейтральных metadata.

### Экспертная оценка

Назначение: дать эксперту место для проверки результата интеллектуального анализа.

Содержит:

- элементы карты влияния, доступные для проверки;
- отметки по элементам: подтверждено, исправлено, отклонено, требует уточнения;
- комментарии эксперта к элементам;
- текст исправления, если элемент скорректирован;
- пропущенные элементы, которые эксперт добавляет вручную;
- дополнительные исправления;
- оценку достаточности контекста;
- оценку полезности результата;
- общий комментарий эксперта;
- сохранение экспертной оценки.

Логика экрана:

- экспертная оценка не должна смешиваться с исходным AI/RAG/LLM-выводом;
- эксперт может подтвердить полезность результата, но также может зафиксировать ошибки, пропуски и спорные элементы;
- оценка достаточности контекста особенно важна для external AI/RAG режима, если retrieved context частичный или недоступный;
- сохраненная оценка является основанием для последующего экспертного заключения.

### Экспертное заключение

Назначение: зафиксировать итоговое экспертное заключение человека.

Содержит:

- тип заключения;
- комментарий;
- обоснование;
- дату фиксации;
- статус артефакта после фиксации.

Варианты заключения:

- принять;
- принять с ограничениями;
- отправить на уточнение;
- разделить на несколько задач;
- отклонить;
- вернуть на повторный анализ.

Логика экрана:

- заключение фиксирует человек;
- варианты заключения не создают задачи, не запускают workflow и не выполняют действий во внешних системах;
- обоснование должно быть достаточно явным для последующего обсуждения, экспорта и экспериментальной проверки;
- повторный анализ после экспертной фиксации должен рассматриваться с учетом правил воспроизводимости результата.

### Демо-пример

Назначение: показать пользователю и исследователю воспроизводимый пример полного сценария без реальных корпоративных секретов и конфиденциальных данных.

Содержит:

- обезличенное текущее состояние;
- обезличенное проектное изменение;
- описание ситуации и источник;
- несколько ручных фрагментов контекста;
- запуск через `Direct LLM` с demo/mock provider или через заранее настроенный external AI/RAG контур без раскрытия ключей;
- предварительную карту влияния;
- отображение оснований и ограничений;
- экспертную оценку с несколькими подтвержденными, исправленными и требующими уточнения элементами;
- экспертное заключение;
- Markdown/JSON экспорт.

Логика экрана или сценария:

- демо-пример должен быть пригоден для ручной проверки полного пути;
- default build/test не должны требовать внешней сети, Dify, DeepSeek, API keys, user-secrets или локально запущенных внешних сервисов;
- если используется external AI/RAG demo, оно должно быть явно помечено как зависящее от внешне настроенного контура;
- демо не должно превращаться в production seed с реальными данными.

## Нежелательные расширения MVP-3

MVP-3 UI flow не должен добавлять:

- task board, backlog, sprint board или назначение исполнителей;
- approval workflow, маршрутизацию согласования или эскалации;
- автоматическое создание задач из экспертного заключения;
- автоматическое изменение требований, документов, API или кода;
- автоматическое создание pull request;
- корпоративную wiki или замену Confluence;
- собственный RAG pipeline, embeddings, rerank или vector database;
- прямую зависимость UI от Dify-specific DTO или provider payload;
- отображение LLM/AI/RAG результата как управленческого решения;
- PDF export как обязательный сценарий MVP-3.

## Проверка соответствия текущему приложению

Документ согласуется с текущей структурой web/UI на уровне чтения:

- список анализов соответствует странице сохраненных анализов;
- создание анализа соответствует текущему вводу названия, исходного описания, проектного запроса, ситуации и источника;
- проверка ввода уже содержит выбор `Direct LLM` или `External AI/RAG`;
- детали анализа уже являются центральным местом просмотра входных данных, результата, карты влияния, retrieved context, экспертного заключения и экспорта;
- экспертная оценка уже отделена от экспертного заключения;
- Markdown/JSON экспорт уже смыслово привязан к зафиксированному экспертному заключению.

Пункты "тип проектного запроса" и "текущее состояние" в этом документе описаны как целевой смысловой слой MVP-3. Они не требуют немедленного изменения модели, страниц или экспорта без отдельного прохождения workflow, требований, технического проекта, implementation plan и явного решения о реализации конкретной task.
