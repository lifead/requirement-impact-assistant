# Черновик требований MVP-3 к программе анализа влияния проектных запросов

## Назначение документа

Документ фиксирует черновик требований MVP-3 к программе анализа влияния проектных запросов.

Документ является этапом requirements-draft. Он не является техническим проектом, implementation plan, task breakdown, описанием архитектуры, схемой базы данных, описанием endpoint-ов, UI-макетом, JSON schema, prompt specification или выбором конкретного provider-а.

Назначение документа - зафиксировать требования к поведению программы на уровне входной формы анализа, типизации проектного запроса, демо-сценария, страницы результата, основания и ограничений результата, экспертной оценки и регистрационной готовности.

MVP-3 рассматривается как развитие уже зафиксированной линии MVP-0, MVP-1 и MVP-2:

- MVP-0 зафиксировал карточку анализа, ручной контекст, direct LLM analysis, экспертную оценку, экспертное заключение, Markdown/JSON export и воспроизводимость результата.
- MVP-1 зафиксировал два режима анализа: `DirectLlm` и `ExternalRag`, различение manual context и retrieved context, сохранение mode/engine/provider/adapter metadata и явное отображение ограничений основания результата.
- MVP-2 зафиксировал Dify Agent как конкретный внешний provider/adapter за границей `ExternalRagAnalysisEngine -> IExternalRagAdapter -> DifyExternalRagAdapter`, не превращая Dify в публичную модель программы.

MVP-3 не отменяет эти решения и не смешивает предварительный AI/RAG/LLM результат с экспертным решением человека.

## Основание

Основанием для требований являются:

- `docs/workflow/README.md`;
- `docs/workflow/checklists.md`;
- `docs/workflow/development-cycle.md`;
- документы MVP-0 в `docs/program/mvp0/`;
- документы MVP-1 в `docs/program/mvp1/`;
- документы MVP-2 в `docs/program/mvp2/`;
- проектная позиция о том, что AI/RAG/LLM формирует предварительный аналитический материал, а экспертное заключение и управленческое рассмотрение остаются за человеком.

В требования перенесены следующие рамочные решения:

- интеллектуальный анализ должен проходить через application-level boundary, а не через прямые вызовы provider-а из пользовательского сценария;
- Direct LLM и External AI/RAG являются режимами анализа, а не субъектами управления;
- внешний AI/RAG provider или adapter может предоставить retrieved context, metadata, warnings и limitations, но не заменяет экспертное основание итогового заключения;
- manual context задает пользователь в карточке анализа; RAG knowledge base находится во внешнем контуре/provider-е и не является новой базой знаний приложения; retrieved context - это только фактически возвращенные provider-ом фрагменты и/или metadata для конкретного анализа;
- Dify-specific детали не должны становиться публичной моделью требований;
- программа не должна превращаться в Jira, Confluence, GitLab, ALM, RAG-платформу, dashboard, PDF-генератор или workflow-систему.

## Назначение MVP-3

MVP-3 должен уточнить и стабилизировать пользовательский контур анализа проектного запроса так, чтобы программа могла принимать разные типы проектных запросов, формировать понятный входной набор данных, показывать результат в структуре карты влияния и отдельно фиксировать основания, ограничения и экспертную оценку.

MVP-3 должен быть пригоден для демонстрации и для последующей регистрации программы для ЭВМ за счет устойчивого функционального состава:

- создание анализа проектного запроса;
- выбор типа проектного запроса;
- описание текущего состояния и проектного изменения;
- фиксация ситуации, причины и источника изменения;
- запуск выбранного режима интеллектуального анализа;
- просмотр карты влияния и основания результата;
- экспертная оценка человеком;
- фиксация типового экспертного решения.

Программа не должна утверждать изменение, отклонять его, назначать исполнителей, создавать задачи, изменять требования, документы, API, интеграции, архитектурные решения или код.

## Границы MVP-3

В MVP-3 входит:

- единый смысловой набор типов проектного запроса;
- требования к форме создания анализа;
- подсказки к полям формы создания анализа;
- демо-сценарий на подготовленных или обезличенных материалах;
- страница результата анализа с картой влияния и связанными блоками;
- блок оснований и ограничений результата;
- экспертная оценка результата человеком;
- типовые решения эксперта;
- требования к регистрационной готовности.

В MVP-3 не входит:

- новый RAG, собственные embeddings, vector database, rerank или agentic workflow;
- новая интеграция с Jira, Confluence, GitLab, ALM, почтой, чатами или корпоративными workflow-системами;
- dashboard для управления задачами или исследовательскими метриками;
- обязательная генерация PDF;
- автоматическое создание задач, pull request, изменений требований, документов, API или кода;
- техническая фиксация структуры БД, endpoint-ов, UI-компонентов, migrations, appsettings, provider SDK, prompt-ов или response schema;
- превращение Dify или другого provider-а в обязательную публичную модель программы.

## Типы проектного запроса

Программа должна позволять пользователю указать тип проектного запроса.

Минимальный список типов проектного запроса:

- Изменение требования;
- Новая функциональность;
- Исправление дефекта;
- Уточнение требования;
- Изменение API / интеграции;
- Изменение архитектурного ограничения;
- Изменение проектного решения;
- Изменение пользовательского сценария;
- Другое.

Тип проектного запроса должен помогать структурировать входные данные и результат анализа, но не должен автоматически определять экспертное решение или запускать workflow.

Если выбран тип "Другое", программа должна позволять пользователю описать смысл запроса текстом.

## Форма создания анализа

Форма создания анализа должна поддерживать минимальный набор полей:

- Название анализа;
- Тип проектного запроса;
- Текущее состояние;
- Проектное изменение;
- Ситуация и причина изменения;
- Источник изменения.

Поле "Название анализа" должно позволять кратко идентифицировать анализ в списке сохраненных артефактов.

Поле "Тип проектного запроса" должно использовать минимальный список типов из настоящего документа.

Поле "Текущее состояние" должно описывать, как сейчас устроено требование, функция, API, интеграция, архитектурное ограничение, проектное решение или пользовательский сценарий, который меняется.

Поле "Проектное изменение" должно описывать, что именно предлагается изменить, добавить, уточнить или исправить.

Поле "Ситуация и причина изменения" должно фиксировать обстоятельства возникновения запроса, бизнес-причину, техническую причину, дефект, уточнение или иную мотивацию изменения.

Поле "Источник изменения" должно фиксировать происхождение запроса: участник проекта, обсуждение, документ, внешний запрос, результат тестирования, архитектурное ограничение, демонстрационный кейс или иной источник.

Связанный проектный контекст может быть добавлен пользователем как manual context или частично найден во внешней RAG knowledge base при соответствующем режиме анализа. Отсутствие найденного, полного или доступного контекста не должно скрываться в результате.

## Подсказки к полям

Форма создания анализа должна содержать понятные подсказки, которые помогают пользователю заполнить поля без смешения текущего состояния, предлагаемого изменения и экспертного вывода.

Минимальные подсказки:

- Название анализа: "Кратко назовите анализ так, чтобы его можно было отличить от других сохраненных анализов."
- Тип проектного запроса: "Выберите ближайший тип изменения. Если ни один тип не подходит, выберите `Другое` и поясните запрос в текстовых полях."
- Текущее состояние: "Кратко опишите, как сейчас работает требование, функция, API или проектное решение, которое меняется. Связанный проектный контекст может быть частично найден во внешней RAG knowledge base при соответствующем режиме анализа; найденные материалы отображаются как retrieved context с ограничениями."
- Проектное изменение: "Опишите предлагаемое изменение: что нужно добавить, изменить, уточнить, исправить или пересмотреть."
- Ситуация и причина изменения: "Опишите, почему возник запрос: новая потребность, дефект, уточнение, ограничение, изменение интеграции, решение эксперта или другая причина."
- Источник изменения: "Укажите, откуда поступил запрос или где он был зафиксирован: документ, обсуждение, задача, тестирование, эксперт, демонстрационный кейс или другой источник."

Подсказки не должны обещать полноту внешнего поиска или гарантировать, что внешний AI/RAG-контур найдет весь проектный контекст.

## Демо-сценарий

MVP-3 должен поддерживать демонстрационный сценарий без фиксации конкретной реализации.

Демо-сценарий должен позволять показать:

1. Создание анализа проектного запроса.
2. Выбор одного из минимальных типов проектного запроса.
3. Заполнение текущего состояния, проектного изменения, ситуации и причины, источника изменения.
4. Запуск доступного режима анализа через согласованную границу интеллектуального анализа.
5. Получение структурированного результата с картой влияния.
6. Просмотр основания результата: режим анализа, provider/adapter, retrieved context или ограничение его отсутствия, найденные источники и warnings.
7. Просмотр противоречий, недостающей информации, вопросов и рисков.
8. Фиксацию экспертной оценки и типового решения эксперта.

Демо-сценарий должен использовать подготовленные, обезличенные или специально созданные материалы. Он не должен требовать реальных корпоративных данных, реальной Jira/Confluence/GitLab-интеграции, нового RAG-пайплайна, обязательного PDF или обязательного сетевого вызова.

## Страница результата

Страница результата должна показывать результат интеллектуального анализа как предварительный аналитический материал, пригодный для экспертного рассмотрения.

Страница результата должна включать следующие блоки:

- карта влияния проектного запроса;
- сводка проектного запроса и предполагаемого изменения;
- затронутые требования;
- затронутые API и интеграции;
- затронутые документы;
- затронутые тесты;
- затронутые проектные решения;
- затронутые архитектурные и организационные ограничения;
- противоречия;
- недостающая информация;
- вопросы для уточнения;
- риски;
- варианты экспертного рассмотрения.

Блоки результата должны быть представлены так, чтобы пользователь мог отличить:

- исходные данные анализа;
- preliminary AI/RAG/LLM результат;
- найденные или использованные основания результата;
- ограничения результата;
- экспертную оценку человека;
- итоговое экспертное решение.

Варианты экспертного рассмотрения не должны отображаться как автоматические управленческие решения AI/RAG/LLM.

## Блок оснований и ограничений результата

Страница результата должна содержать отдельный блок "Основания и ограничения результата" или смысловой аналог.

Блок должен включать:

- режим анализа;
- provider/adapter или сведения об их недоступности;
- retrieved context state;
- retrieved context items или metadata, если они доступны;
- найденные источники;
- сведения о том, был ли найденный контекст полным, частичным, metadata-only или недоступным;
- ограничения воспроизводимости результата;
- warnings и ограничения анализа;
- сведения о неполноте или спорности результата, если они есть.

Для direct LLM режима программа не должна создавать искусственный retrieved context. В таком случае блок оснований должен показывать, что результат основан на исходных данных анализа и доступном manual context, если он был использован.

Для external AI/RAG режима программа должна показывать retrieved context или явно фиксировать, что он недоступен, неполон или представлен только metadata.

Retrieved context и найденные источники являются основанием предварительного AI/RAG результата, но не являются экспертным основанием итогового решения.

Ограничения воспроизводимости должны быть видимы пользователю и не должны маскироваться под экспертное заключение.

Полный raw response provider-а, provider-specific DTO, секреты, токены, cookies, CSRF, bearer values и чувствительные diagnostic payloads не должны становиться обязательной частью результата, пользовательской модели или регистрационного описания программы.

## Экспертная оценка

Программа должна позволять человеку выполнить экспертную оценку результата интеллектуального анализа.

Экспертная оценка должна поддерживать:

- оценку полноты и применимости результата;
- подтверждение элементов карты влияния;
- корректировку элементов карты влияния;
- отклонение спорных или ошибочных элементов;
- фиксацию пропущенных элементов;
- оценку достаточности контекста;
- комментарий или обоснование эксперта;
- отделение экспертной оценки от AI/RAG/LLM результата.

Экспертная оценка не должна редактировать retrieved context как manual context и не должна превращать найденные источники внешнего provider-а в итоговое управленческое основание без проверки человеком.

## Типовые решения эксперта

Программа должна позволять эксперту зафиксировать типовое решение из минимального набора:

- принять;
- принять с ограничениями;
- отправить на уточнение;
- разделить на несколько задач;
- отклонить;
- вернуть на повторный анализ.

Типовое решение фиксируется человеком. Программа не должна автоматически выбирать типовое решение, создавать задачи, маршрутизировать согласование, назначать исполнителей или выполнять действия во внешних системах.

Для решения "вернуть на повторный анализ" программа должна сохранять смысл решения как экспертную фиксацию необходимости повторной аналитической подготовки после уточнения данных или контекста. Это не должно означать автоматический запуск нового анализа без действия пользователя.

Для решения "разделить на несколько задач" программа должна сохранять смысл решения как экспертный вывод. Это не должно означать автоматическое создание задач или workflow.

## Регистрационная готовность

MVP-3 должен поддерживать регистрационную готовность программы для ЭВМ на уровне устойчивого описания функционального состава.

Для регистрационной готовности существенны:

- понятное назначение программы как ассистента анализа влияния проектных запросов;
- фиксируемые входные данные анализа;
- минимальный классификатор типов проектного запроса;
- структурированный результат в форме карты влияния;
- отдельный блок оснований и ограничений результата;
- явное разделение предварительного AI/RAG/LLM результата и экспертного решения человека;
- сохранение экспертной оценки и типового решения эксперта;
- пригодность демонстрационного сценария на обезличенных или подготовленных данных;
- отсутствие обязательной зависимости от конкретного внешнего provider-а как от публичной сущности программы;
- отсутствие функций автоматического управления задачами, согласованиями, требованиями, API, документами или кодом.

Регистрационное описание не должно представлять программу как самостоятельный субъект управления, RAG-платформу, Jira/Confluence/GitLab-замену, workflow-платформу или инструмент автоматического принятия решений.

## Функциональные требования

FR-1. Программа должна позволять создать анализ проектного запроса с минимальными полями, указанными в разделе "Форма создания анализа".

FR-2. Программа должна позволять выбрать тип проектного запроса из минимального списка типов.

FR-3. Программа должна поддерживать тип "Другое" для запросов, которые не укладываются в минимальный список.

FR-4. Программа должна показывать подсказки к полям формы создания анализа.

FR-5. Программа должна позволять запустить анализ по заполненным входным данным через согласованную границу интеллектуального анализа.

FR-6. Программа должна формировать результат в структуре карты влияния и связанных блоков страницы результата.

FR-7. Программа должна показывать сводку проектного запроса и предполагаемого изменения.

FR-8. Программа должна показывать затронутые требования, API, интеграции, документы, тесты, проектные решения и ограничения.

FR-9. Программа должна показывать противоречия, недостающую информацию, вопросы для уточнения и риски.

FR-10. Программа должна показывать варианты экспертного рассмотрения как предварительный аналитический материал, а не как решение программы.

FR-11. Программа должна показывать отдельный блок оснований и ограничений результата.

FR-12. Программа должна показывать режим анализа и сведения о provider/adapter на уровне, достаточном для воспроизводимого рассмотрения результата.

FR-13. Программа должна показывать retrieved context, найденные источники или ограничения их отсутствия, если они применимы к выбранному режиму анализа.

FR-14. Программа должна явно показывать ограничения воспроизводимости и warnings, если они есть.

FR-15. Программа должна позволять эксперту оценить результат интеллектуального анализа отдельно от самого результата.

FR-16. Программа должна позволять эксперту зафиксировать типовое решение из минимального набора.

FR-17. Программа должна сохранять комментарий или обоснование эксперта как часть экспертной фиксации.

FR-18. Программа не должна смешивать AI/RAG/LLM результат, retrieved context, warnings и экспертное решение так, будто они имеют один источник.

FR-19. Программа должна поддерживать демонстрационный сценарий на подготовленных или обезличенных материалах.

FR-20. Программа должна сохранять терминологию, в которой AI/RAG/LLM формирует предварительный аналитический материал, а человек принимает и фиксирует экспертное решение.

## Нефункциональные требования

NFR-1. Требования MVP-3 должны сохранять совместимость с проектной позицией о разделении аналитической роли AI/RAG/LLM и управленческой роли человека.

NFR-2. Результат анализа должен быть пригоден для воспроизводимого экспертного рассмотрения: пользователь должен видеть исходные данные, режим анализа, доступное основание результата, ограничения и экспертную фиксацию.

NFR-3. Ограничения результата, неполнота retrieved context, warnings, недостающая информация и спорные выводы не должны скрываться.

NFR-4. Демо-сценарий должен быть воспроизводимым на подготовленных или обезличенных материалах без обязательных внешних корпоративных интеграций.

NFR-5. Требования не должны фиксировать конкретный стек, структуру хранения, endpoint-ы, migrations, UI-компоненты, provider-specific DTO, prompt-ы или response schema.

NFR-6. Требования не должны вводить новую RAG-платформу, embeddings, dashboard, PDF, Jira/Confluence/GitLab integration или workflow-платформу.

NFR-7. Описание функциональности должно быть пригодно для регистрационного описания программы для ЭВМ без представления программы как субъекта управления.

## Открытые вопросы

Следующие вопросы можно перенести в последующие этапы workflow без изменения смысла требований:

- как технически хранить расширенный набор типов проектного запроса;
- как именно расположить блоки страницы результата в конкретном UI;
- какие технические поля provider/adapter metadata попадут в сохраненную модель и экспорт;
- как формально версионировать подсказки к полям;
- как оформлять демонстрационный набор данных для конкретного показа или экспериментальной проверки.

Эти вопросы должны рассматриваться после review требований и явного решения о переходе к техническому проектированию или implementation planning.

## Что пока не фиксируется

На этапе requirements-draft MVP-3 не фиксируются:

- структура базы данных;
- объектная модель;
- API-контракты;
- endpoint-ы;
- migrations;
- appsettings;
- конкретные UI-компоненты и визуальный дизайн;
- конкретная JSON schema;
- конкретные prompt-ы или response schema;
- конкретный provider SDK;
- новая внешняя интеграция;
- новый RAG, embeddings, rerank, vector database или agentic workflow;
- dashboard;
- PDF export;
- task/workflow automation;
- implementation plan;
- тестовые сценарии;
- commit plan.

Эти вопросы должны рассматриваться только после review требований и отдельного явного решения о следующем этапе.
