# MVP-3 demo smoke checklist

## Назначение

Этот checklist описывает воспроизводимый ручной показ MVP-3 на обезличенных данных. Он является только документацией: не добавляет seed data, route, UI button, browser automation, production fixture, секреты или обязательные внешние зависимости.

Default smoke выполняется offline:

- `Direct LLM` - через настроенный demo/mock LLM provider приложения.
- `External AI/RAG` - через `MockExternalRagAdapter`, если real external adapter не настроен.
- Real Dify, DeepSeek, сеть, user-secrets и API keys не являются gate для этого checklist.

## Обезличенный кейс

Сценарий: в демонстрационной системе обработки заявок меняется правило уведомления заявителя после изменения статуса заявки.

Данные не относятся к реальной организации, проекту, пользователям, API или документам.

### Input data

| Слой | Значение для ввода |
| --- | --- |
| Название | `Demo MVP-3: уведомление заявителя после изменения статуса заявки` |
| Тип проектного запроса | `Изменение API / интеграции` |
| Текущее состояние | `В демонстрационной системе заявок оператор вручную меняет статус заявки. После изменения статуса заявитель видит новое состояние только при повторном входе в личный кабинет. Внешняя система уведомлений сейчас получает события только о создании заявки.` |
| Проектное изменение | `Добавить отправку события во внешнюю систему уведомлений при переходе заявки в статусы "Принята в работу", "Требует уточнения" и "Закрыта". Событие должно содержать обезличенный идентификатор заявки, новый статус, дату изменения и код причины без персональных данных.` |
| Ситуация и причина изменения | `По итогам пилотной эксплуатации пользователи поздно узнают о смене статуса. Требуется повысить прозрачность процесса без расширения состава персональных данных в интеграции.` |
| Источник изменения | `Протокол демонстрационного обсуждения DEMO-REQ-17 от 2026-06-01` |

### Manual context fragments

Добавить в карточку анализа два ручных фрагмента контекста:

| Тип | Источник | Текст |
| --- | --- | --- |
| `Описание API` | `DEMO-API-NOTIFY-v1` | `Внешняя система уведомлений принимает событие status_changed с полями requestRef, statusCode, changedAt, reasonCode. Поля ФИО, телефон, email и адрес в событие не передаются.` |
| `Архитектурное ограничение` | `DEMO-ADR-04` | `Интеграция с уведомлениями должна быть асинхронной. Ошибка отправки события не должна блокировать сохранение нового статуса заявки, но должна фиксироваться для последующей проверки.` |

## Что должно быть различимо в показе

| Слой | Что показать | Чем не является |
| --- | --- | --- |
| Current state / текущее состояние | Входной снимок того, как работает демонстрационная система до изменения. | Не RAG knowledge base и не retrieved context. |
| Project change / проектное изменение | Предмет анализа: что предлагается изменить. | Не принятое управленческое решение. |
| Manual context | Материалы, которые пользователь явно добавил в карточку анализа. | Не найденные внешним provider-ом источники. |
| Retrieved context | Только сохраненные фрагменты или metadata, которые вернул external AI/RAG adapter для конкретного анализа. | Не ручной контекст и не доказательство правильности результата само по себе. |
| Preliminary result | Карта влияния и preliminary analytical material, сформированные AI/RAG/LLM. | Не экспертное заключение и не автоматическое поручение к реализации. |
| Expert evaluation | Человеческая проверка элементов карты влияния: подтверждения, исправления, пропуски, достаточность контекста и полезность результата. | Не повторный вызов provider-а. |
| Expert decision / expert conclusion | Зафиксированное человеком заключение с типом, комментарием, обоснованием и датой. | Не workflow action, не taskboard, не автоматический reanalysis. |

## Smoke A: Direct LLM через demo/mock provider

1. Открыть список анализов и создать новый анализ.
2. Заполнить поля из раздела `Input data`.
3. Сохранить карточку анализа.
4. На странице деталей добавить оба фрагмента из `Manual context fragments`.
5. Открыть проверку ввода перед анализом.
6. Выбрать режим `Direct LLM`.
7. Проверить, что страница проверки показывает provider-neutral пояснение и не выполняет health-check внешнего AI/RAG provider-а при просмотре.
8. Запустить предварительный анализ.
9. Вернуться на детали анализа и проверить смысловую навигацию:
   - `Входные данные` показывают тип проектного запроса, текущее состояние, проектное изменение, ситуацию и источник.
   - `Manual context` показывает добавленные пользователем фрагменты.
   - `Retrieved context` для `Direct LLM` показывает, что отдельный retrieved context не создается.
   - `Предварительный результат` содержит карту влияния как preliminary analytical material.
   - `Основания и ограничения` показывают режим анализа, metadata результата, warnings или limitation notes, если они есть.
10. Открыть экспертную оценку.
11. Проверить, что экран экспертной оценки показывает входные данные, preliminary impact map, состояние retrieved context и warnings/limitations без повторного запуска анализа.
12. Зафиксировать пример экспертной оценки:
    - один элемент карты отметить как `Подтверждено`;
    - один элемент отметить как `Требует уточнения` с комментарием `Нужно уточнить перечень статусов, для которых событие обязательно`;
    - добавить пропущенный элемент `Тестовый сценарий отправки события при ошибке внешней системы`;
    - указать достаточность контекста `Частично достаточный`;
    - указать полезность результата `Частично полезен`.
13. Открыть экспертное заключение.
14. Зафиксировать тип `Принять с ограничениями` с комментарием `Предварительный анализ полезен, но требуется уточнить retry-поведение и тестовые сценарии` и обоснованием `Эксперт подтвердил ключевые затронутые элементы, но добавил недостающую проверку отказа внешней системы`.
15. Вернуться на детали анализа.
16. Скачать Markdown и JSON export.
17. Проверить в export, что сохранены разделы input, preliminary result, grounds/retrieved context, expert evaluation, expert conclusion и decision boundary; export не должен запускать повторный анализ.

Ожидаемый результат Direct LLM smoke: полный путь проходит без Dify, DeepSeek, сети и user-secrets. Retrieved context не подменяется искусственными источниками.

## Smoke B: External AI/RAG через mock adapter

1. Использовать тот же обезличенный кейс. Для чистоты показа можно создать второй анализ с теми же входными данными и пометкой в названии `External AI/RAG mock`.
2. Добавить те же manual context fragments.
3. Открыть проверку ввода перед анализом.
4. Выбрать режим `External AI/RAG`.
5. Проверить, что описание режима говорит о mock fallback или внешнем adapter-е в зависимости от конфигурации приложения, но не раскрывает endpoint, API key, bearer token, cookies, CSRF или raw provider payload.
6. Запустить предварительный анализ.
7. На деталях анализа проверить:
   - режим результата отображается как `External AI/RAG`;
   - adapter metadata безопасна и provider-neutral;
   - manual context отделен от retrieved context;
   - retrieved context показывает фактически сохраненное состояние mock adapter-а: available, partial, metadata-only или unavailable;
   - если mock adapter не вернул фрагменты, отсутствие retrieved context показано как limitation, а не замаскировано выдуманными источниками.
8. Открыть экспертную оценку и проверить, что retrieved context state, warnings count и limitation notes видны эксперту до оценки.
9. Зафиксировать экспертную оценку с акцентом на достаточность контекста:
   - если retrieved context available или partial, отметить, какие элементы карты подтверждаются ручным и найденным контекстом;
   - если retrieved context unavailable или metadata-only, указать `Частично достаточный` или `Недостаточный` и пояснить ограничение воспроизводимости.
10. Зафиксировать экспертное заключение человеком. Для демонстрации подходит `Требуется уточнение` или `Принять с ограничениями`, если retrieved context неполный.
11. Скачать Markdown и JSON export.
12. Проверить, что export отражает analysis mode, retrieved context state/items или их отсутствие, preliminary material и человеческое expert conclusion как разные слои.

Ожидаемый результат External AI/RAG mock smoke: сценарий остается offline/default-safe и не требует real Dify. Mock adapter используется только как существующий application-level fallback, без новой RAG feature.

## Optional manual: real Dify smoke вне default gate

Этот раздел не является обязательной проверкой MVP-3 и не должен блокировать review, build, tests или demo checklist.

Real Dify smoke допустим только вручную, в отдельной локальной конфигурации с обезличенной демонстрационной knowledge base и секретами вне репозитория. В документах, export и screenshots нельзя фиксировать реальные ключи, bearer values, cookies, CSRF, private endpoint с чувствительными параметрами, персональные данные или raw provider payload.

Минимальные правила optional smoke:

- использовать только placeholders в заметках: `<DIFY_ENDPOINT>`, `<DIFY_APP_ID>`, `<DIFY_API_KEY_IN_USER_SECRETS>`;
- не коммитить `appsettings`, user-secrets, локальную БД с приватными данными или export с секретами;
- не считать отказ сети или Dify причиной провала default MVP-3 smoke;
- не расширять сценарий до настройки нового provider-а, нового RAG pipeline, embeddings, rerank, vector database или workflow.

## Review checklist для документа

- [ ] Документ содержит один реалистичный обезличенный кейс.
- [ ] Direct LLM и External AI/RAG mock описаны как offline/default-safe варианты.
- [ ] Real Dify упомянут только как optional manual scenario вне default gate.
- [ ] Current state, project change, manual context, retrieved context, preliminary result, expert evaluation и expert decision различены явно.
- [ ] Нет secrets, private data, bearer values, cookies, CSRF, real provider payload или чувствительных endpoint-ов.
- [ ] Нет production seed data, route, UI button, browser automation, code/test changes или обязательной сети.
- [ ] Документ не предлагает новую integration/RAG/workflow feature.
